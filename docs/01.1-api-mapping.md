## 1. Convenções gerais

### Autenticação

Endpoints marcados como **🔒 Autenticação obrigatória** exigem o header:
```
Authorization: Token eyJhbGciOiJIUzUxMiJ9...
```

Endpoints marcados como **🔓 Autenticação opcional** funcionam sem token, mas retornam dados enriquecidos (ex: `following: true`, `favorited: true`) quando autenticado.

Endpoints marcados como **🌐 Público** não requerem autenticação.

### Content-Type

Todas as requisições com body devem incluir:
```
Content-Type: application/json
```

Todas as respostas retornam:
```
Content-Type: application/json; charset=utf-8
```

### Paginação REST

Os endpoints de listagem aceitam:

| Parâmetro | Tipo | Padrão | Descrição |
|---|---|---|---|
| `limit` | integer | 20 | Número máximo de itens retornados |
| `offset` | integer | 0 | Número de itens a pular |

### Paginação GraphQL

O schema usa **cursor-based pagination** (padrão Relay) — ver [Seção 5](#5-cursor-pagination--graphql) para detalhes completos.

| Argumento | Tipo | Direção |
|---|---|---|
| `first` | Int | Navega para frente — N itens após `after` |
| `after` | String | Cursor de início (exclusivo) para paginação forward |
| `last` | Int | Navega para trás — N itens antes de `before` |
| `before` | String | Cursor de fim (exclusivo) para paginação backward |

🔓 Autenticação opcional

**Query params:**

| Parâmetro REST | Tipo | Equivalente GraphQL | Descrição |
|---|---|---|---|
| `tag` | string | `withTag` | Filtrar por tag (ex: `?tag=dragons`) |
| `author` | string | `authoredBy` | Filtrar por autor (ex: `?author=jake`) |

| `limit` | integer | `first` | Padrão: `20` |
| `offset` | integer | via cursor | Padrão: `0` |

> **Nota:** os nomes dos parâmetros de filtro diferem entre REST e GraphQL. REST usa `author`, `tag`, `favorited`; GraphQL usa `authoredBy`, `withTag`, `favoritedBy`. O comportamento é equivalente.

## 3. GraphQL API — Schema completo

Endpoint único: `POST http://localhost:8080/graphql`

Requisições enviadas como:
```json
{
  "query": "query { ... }",
  "variables": {}
}
```

> **Nota técnica:** o pacote `io.spring.graphql` que implementa os resolvers é **gerado automaticamente** pelo DGS Codegen a partir do arquivo `schema.graphqls` — ver ADR-004. Nunca editar manualmente os arquivos deste pacote.

---

### 3.1 Queries

```graphql
type Query {
  # Retorna um artigo pelo slug
  article(slug: String!): Article

  # Lista artigos com cursor pagination e filtros opcionais
  # Nota: nomes dos argumentos diferem dos query params REST
  articles(
    first: Int          # equivalente REST: limit
    after: String       # cursor de início
    last: Int
    before: String      # cursor de fim
    authoredBy: String  # equivalente REST: ?author=
    favoritedBy: String # equivalente REST: ?favorited=
    withTag: String     # equivalente REST: ?tag=
  ): ArticlesConnection

  # Retorna o usuário autenticado atual
  me: User

  # Feed do usuário autenticado (artigos de quem ele segue)
  feed(first: Int, after: String, last: Int, before: String): ArticlesConnection

  # Retorna o perfil de um usuário
  # O tipo Profile retornado inclui sub-campos articles, favorites, feed
  profile(username: String!): ProfilePayload

  # Lista todas as tags disponíveis
  tags: [String]
}
```

**Exemplos de uso:**

```graphql
# Buscar artigo com comentários paginados
query {
  article(slug: "how-to-train-your-dragon") {
    title
    body
    readingTimeMinutes  # disponível após Fase 5
    author { username bio }
    comments(first: 10) {
      edges {
        node { id body createdAt author { username } }
      }
      pageInfo { hasNextPage endCursor }
    }
  }
}

# Listar artigos por tag com cursor pagination
query {
  articles(first: 10, withTag: "dragons") {
    edges {
      cursor
      node { slug title description favoritesCount }
    }
    pageInfo { hasNextPage endCursor }
  }
}

# Perfil com artigos favoritados pelo usuário
query {
  profile(username: "jake") {
    profile {
      username
      bio
      following
      favorites(first: 5) {
        edges { node { slug title } }
        pageInfo { hasNextPage endCursor }
      }
    }
  }
}
```

### 3.2 Mutations

```graphql
type Mutation {
  # --- Usuário & Perfil ---

  # Cria novo usuário — retorna union UserResult (sucesso ou erro estruturado)
  createUser(input: CreateUserInput): UserResult

  # Login — retorna UserPayload diretamente (sem union)
  login(password: String!, email: String!): UserPayload

  # Atualiza dados do usuário autenticado
  updateUser(changes: UpdateUserInput!): UserPayload

  # Segue um usuário
  followUser(username: String!): ProfilePayload

  # Deixa de seguir um usuário
  unfollowUser(username: String!): ProfilePayload

  # --- Artigo ---

  # Cria artigo (calcula readingTimeMinutes automaticamente após Fase 5)
  createArticle(input: CreateArticleInput!): ArticlePayload

  # Atualiza artigo (recalcula readingTimeMinutes se body mudar após Fase 5)
  updateArticle(slug: String!, changes: UpdateArticleInput!): ArticlePayload

  # Favorita artigo
  favoriteArticle(slug: String!): ArticlePayload

  # Desfavorita artigo
  unfavoriteArticle(slug: String!): ArticlePayload

  # Deleta artigo — após Fase 5: soft delete (is_deleted = true)
  deleteArticle(slug: String!): DeletionStatus

  # --- Comentário ---

  # Adiciona comentário
  addComment(slug: String!, body: String!): CommentPayload

  # Deleta comentário — após Fase 5: soft delete
  deleteComment(slug: String!, id: ID!): DeletionStatus
}
```

**Exemplo — criar usuário com tratamento de union:**
```graphql
mutation {
  createUser(input: {
    email: "jake@jake.jake"
    username: "jake"
    password: "jakejake"
  }) {
    ... on UserPayload {
      user { email username token }
    }
    ... on Error {
      message
      errors { key value }
    }
  }
}
```

**Exemplo — criar artigo:**
```graphql
mutation {
  createArticle(input: {
    title: "How to train your dragon"
    description: "Ever wonder how?"
    body: "You have to believe"
    tagList: ["dragons", "training"]
  }) {
    article {
      slug
      title
      createdAt
      readingTimeMinutes  # retornado automaticamente após Fase 5
    }
  }
}
```

## 4. Mapeamento REST ↔ GraphQL

### Operações principais

| Operação | REST | GraphQL | Auth |
|---|---|---|---|
| Login | `POST /users/login` | `mutation login` | 🌐 |
| Registro | `POST /users` | `mutation createUser` | 🌐 |
| Ver usuário atual | `GET /user` | `query me` | 🔒 |
| Atualizar usuário | `PUT /user` | `mutation updateUser` | 🔒 |
| Ver perfil | `GET /profiles/:username` | `query profile` | 🔓 |
| Seguir usuário | `POST /profiles/:username/follow` | `mutation followUser` | 🔒 |
| Deixar de seguir | `DELETE /profiles/:username/follow` | `mutation unfollowUser` | 🔒 |
| Listar artigos | `GET /articles` | `query articles` | 🔓 |
| Feed personalizado | `GET /articles/feed` | `query feed` | 🔒 |
| Ver artigo | `GET /articles/:slug` | `query article` | 🌐 |
| Criar artigo | `POST /articles` | `mutation createArticle` | 🔒 |
| Atualizar artigo | `PUT /articles/:slug` | `mutation updateArticle` | 🔒 |
| Deletar artigo | `DELETE /articles/:slug` | `mutation deleteArticle` | 🔒 |
| Listar comentários | `GET /articles/:slug/comments` | via `article { comments }` | 🔓 |
| Criar comentário | `POST /articles/:slug/comments` | `mutation addComment` | 🔒 |
| Deletar comentário | `DELETE /articles/:slug/comments/:id` | `mutation deleteComment` | 🔒 |
| Favoritar | `POST /articles/:slug/favorite` | `mutation favoriteArticle` | 🔒 |
| Desfavoritar | `DELETE /articles/:slug/favorite` | `mutation unfavoriteArticle` | 🔒 |
| Listar tags | `GET /tags` | `query tags` | 🌐 |

### Diferenças importantes REST vs GraphQL

| Aspecto | REST | GraphQL | Impacto |
|---|---|---|---|
| **Paginação** | `?limit=20&offset=0` | `first/after/last/before` (cursor) | As estratégias são diferentes — clientes precisam adaptar a lógica de paginação |
| **Filtros de artigos** | `?author=` `?tag=` `?favorited=` | `authoredBy` `withTag` `favoritedBy` | Nomes diferentes — não intercambiáveis |
| **Seleção de campos** | Retorna objeto completo sempre | Retorna apenas os campos solicitados | GraphQL é mais eficiente em banda |
| **Comentários** | Endpoints próprios (`/comments`) | Sub-campo de `Article` (`article { comments }`) | Estrutura diferente de acesso |
| **Artigos por perfil** | Não existe endpoint direto | `profile { articles }` `profile { favorites }` | REST não tem equivalente para esses sub-campos |
| **Erro de createUser** | `{ "errors": { "email": [...] } }` | `union UserResult = UserPayload \| Error` | GraphQL exige tratamento explícito do union |
| **Delete response** | `204 No Content` (sem body) | `DeletionStatus { success: Boolean! }` | REST não retorna dados; GraphQL retorna confirmação |

### Capacidades exclusivas do GraphQL (sem equivalente REST)

| Capacidade | GraphQL | Observação |
|---|---|---|
| Artigos de um perfil | `profile { articles(...) }` | Não existe `GET /profiles/:username/articles` |
| Favoritos de um perfil | `profile { favorites(...) }` | Não existe `GET /profiles/:username/favorites` |
| Feed de um perfil | `profile { feed(...) }` | Diferente de `GET /articles/feed` (que é do usuário autenticado) |
| Comentários paginados | `article { comments(first: N) }` | REST retorna todos os comentários sem paginação |
| Artigo que um comentário pertence | `comment { article { ... } }` | REST não tem esse caminho inverso |

## 5. Cursor pagination — GraphQL

### Como funciona

O schema usa **cursor-based pagination** (padrão Relay) em todos os campos que retornam listas. O cursor é opaco — um valor gerado pelo servidor que representa a posição de um item na lista. No projeto atual, o cursor é baseado no campo `updatedAt` do artigo (via `DateTimeCursor` — ver ADR-005).

### Padrão de uso — paginação forward

```graphql
# Primeira página
query {
  articles(first: 10) {
    edges { cursor node { slug title } }
    pageInfo {
      hasNextPage
      endCursor    # usar como "after" na próxima chamada
    }
  }
}

# Segunda página
query {
  articles(first: 10, after: "cursor-do-endCursor-anterior") {
    edges { cursor node { slug title } }
    pageInfo { hasNextPage endCursor }
  }
}
```

### Padrão de uso — paginação backward

```graphql
# Última página
query {
  articles(last: 10) {
    edges { cursor node { slug title } }
    pageInfo {
      hasPreviousPage
      startCursor    # usar como "before" na chamada anterior
    }
  }
}
```

### Campos que suportam cursor pagination

| Campo | Tipo retornado | Filtros disponíveis |
|---|---|---|
| `articles` (top-level) | `ArticlesConnection` | `authoredBy`, `favoritedBy`, `withTag` |
| `feed` (top-level) | `ArticlesConnection` | nenhum |
| `article.comments` | `CommentsConnection` | nenhum |
| `profile.articles` | `ArticlesConnection` | nenhum |
| `profile.favorites` | `ArticlesConnection` | nenhum |
| `profile.feed` | `ArticlesConnection` | nenhum |

### PageInfo — campos completos

| Campo | Tipo | Quando usar |
|---|---|---|
| `hasNextPage` | `Boolean!` | Verificar se há mais itens para frente antes de chamar com `after` |
| `hasPreviousPage` | `Boolean!` | Verificar se há mais itens para trás antes de chamar com `before` |
| `endCursor` | `String` | Passar como `after` para obter a próxima página |
| `startCursor` | `String` | Passar como `before` para obter a página anterior |

> **Nota para implementação de testes (EPIC-08/09):** ao escrever testes de integração e E2E que usam paginação GraphQL, sempre verificar `pageInfo.hasNextPage` antes de assumir que todos os dados foram retornados. Nunca assumir que uma única chamada retorna todos os registros.

## 6. Campos planejados — Fase 5

Esta seção documenta as mudanças de contrato que serão introduzidas na Fase 5 do projeto. Nenhum desses campos existe no schema atual.

### 6.1 `readingTimeMinutes` — INI-13

**Tipo:** `Int`  
**Descrição:** tempo estimado de leitura em minutos, calculado como `MAX(1, CEIL(wordCount / 200))`.  
**Afeta:** tipo `Article` e todos os endpoints/queries que retornam artigos.

**Adição ao schema GraphQL:**
```graphql
type Article {
  # ... campos existentes ...
  readingTimeMinutes: Int    # nullable — artigos existentes recebem via lazy update
}
```

**Adição ao response REST:**
```json
{
  "article": {
    "slug": "how-to-train-your-dragon",
    "title": "...",
    "readingTimeMinutes": 3,
    "..."
  }
}
```

**Comportamento por endpoint:**

| Endpoint | Comportamento |
|---|---|
| `POST /articles` | Campo calculado e retornado imediatamente |
| `PUT /articles/:slug` (com body novo) | Campo recalculado e retornado |
| `PUT /articles/:slug` (sem body) | Campo inalterado |
| `GET /articles/:slug` (artigo sem campo) | Campo calculado, persistido e retornado (lazy update) |
| `GET /articles` (listagem) | Campo retornado se já calculado; `null` se não (sem lazy update na listagem) |

### 6.2 Soft delete — INI-12

**Mudança no contrato:** nenhuma visível para o consumidor da API.

O `DELETE /articles/:slug` continua retornando `204 No Content`. O artigo continua desaparecendo de todas as listagens. A mudança é **exclusivamente interna** — o registro é marcado como `is_deleted = true` em vez de removido.

**Não existe campo `isDeleted` exposto na API** — é um detalhe de implementação. Se um artigo deletado for solicitado por slug, a resposta continua sendo `404 Not Found`.

> Esta seção serve como referência para testes E2E (EPIC-09 fluxos 4 e 5) e para a documentação OpenAPI que será atualizada na Fase 6 (US-11.08).

## 7. Códigos de resposta e erros

### Códigos HTTP utilizados

| Código | Significado | Quando ocorre |
|---|---|---|
| `200 OK` | Sucesso | GET, PUT, DELETE de favoritos/follow |
| `201 Created` | Criado | POST de criação (usuário, artigo, comentário) |
| `204 No Content` | Sucesso sem body | DELETE de artigo e comentário (mesmo após soft delete) |
| `401 Unauthorized` | Token ausente ou inválido | Endpoints autenticados sem token válido |
| `403 Forbidden` | Sem permissão | Tentar editar/deletar recurso de outro usuário |
| `404 Not Found` | Recurso não encontrado | Slug inválido, username inexistente, artigo deletado |
| `422 Unprocessable Entity` | Erro de validação | Campos obrigatórios faltando ou inválidos |

### Formato de erro REST (`422`)

```json
{
  "errors": {
    "email": ["can't be blank", "is invalid"],
    "password": ["can't be blank"]
  }
}
```

### Formato de erro GraphQL

```json
{
  "errors": [
    {
      "message": "Validation error",
      "locations": [{ "line": 2, "column": 3 }],
      "path": ["createUser"],
      "extensions": {
        "errorType": "BAD_REQUEST"
      }
    }
  ]
}
```

### Erros estruturados GraphQL via union (apenas `createUser`)

```graphql
# createUser retorna union UserResult = UserPayload | Error
# Tratar ambos os casos no cliente:
mutation {
  createUser(input: { ... }) {
    ... on UserPayload {
      user { token username }
    }
    ... on Error {
      message
      errors { key value }
    }
  }
}
```

> **Atenção:** apenas `createUser` usa o union `UserResult`. As mutations `login` e `updateUser` retornam `UserPayload` diretamente — não precisam de tratamento de union.
