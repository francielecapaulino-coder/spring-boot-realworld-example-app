## História de usuário

**Como** engenheiro responsável pela modernização do runtime,
**quero** que todas as referências a Joda-Time sejam substituídas pela API `java.time` —
**para que** a dependência legada seja eliminada e o `ArticleData` fique pronto para virar record (EPIC-06), sem alterar o contrato de API.

---

## Contexto

O codebase usa **Joda-Time** (`org.joda.time.*`). Pontos conhecidos:

- **`ArticleData.java`** usa `org.joda.time.DateTime` no campo `updatedAt` → migrar para `java.time.Instant` (conforme **ADR-005**, antes da conversão para record).
- **`DateTimeCursor`** (cursor de paginação GraphQL) usa Joda-Time → migrar para `java.time`.
- Dependência `joda-time` no `build.gradle` → remover.

**Contrato de API:** o formato de serialização de datas (JSON) **não pode mudar**. A migração `DateTime` → `Instant` deve preservar o formato ISO-8601 atual exposto nos endpoints REST e GraphQL (`docs/01.1-api-mapping.md` v2.0).

### Restrições de arquitetura aplicáveis

- **Nunca usar `import org.joda.time.*`** (AGENTS.md / RATCHET) — usar `java.time.*`.
- **ADR-005:** `ArticleData` implementa `Node` com `getCursor()` explícito; Joda-Time migrado **antes** de records.
- **ADR-004:** não editar `io.spring.graphql.*` (gerado).

---

## O que esta história faz — e o que NÃO faz

### ✅ O que esta história faz

- Substitui todos os usos de `org.joda.time.*` por `java.time.*` (`Instant`/`OffsetDateTime` conforme o caso)
- Migra `ArticleData.updatedAt` e o `DateTimeCursor`
- Remove a dependência `joda-time` do `build.gradle`
- Preserva o formato de serialização de datas (sem mudança de contrato)

### ❌ O que esta história NÃO faz

- Não converte `ArticleData` em record — isso é EPIC-06 (apenas prepara o terreno)
- Não altera o schema GraphQL
- Não muda o contrato de API (formato de datas idêntico)
