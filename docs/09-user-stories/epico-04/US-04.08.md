## História de usuário

**Como** engenheiro responsável pela modernização do runtime,
**quero** habilitar virtual threads no Spring Boot 4 via `spring.threads.virtual.enabled=true` —
**para que** a aplicação escale concorrência de I/O sem reescrever código, confirmando o recurso no log de startup.

---

## Contexto

O **Spring Boot 4** (sobre **Java 25**) suporta **virtual threads** de forma nativa. Basta a propriedade:

```properties
spring.threads.virtual.enabled=true
```

Com isso, o servlet container e executores gerenciados pelo Spring passam a usar virtual threads para requisições, melhorando a vazão de I/O sem mudança de código (sem WebFlux — ver escopo do épico).

A confirmação deve aparecer no **log de startup** e/ou ser observável no comportamento de threads. O critério do marco M3 cita: "Virtual threads habilitados e confirmados no log de startup".

### Restrições de arquitetura aplicáveis

- Mudança em `application.properties`/`.yml` → após alterar, validar startup (`./gradlew bootRun` / `docker compose up`).
- Sem mudança de contrato de API; sem reescrita para WebFlux.

---

## O que esta história faz — e o que NÃO faz

### ✅ O que esta história faz

- Adiciona `spring.threads.virtual.enabled=true` na configuração (perfis aplicáveis)
- Confirma o efeito no log de startup / comportamento de threads
- Mantém todos os testes verdes

### ❌ O que esta história NÃO faz

- Não migra para WebFlux (escopo explicitamente excluído do épico)
- Não altera lógica de negócio nem contrato de API
- Não ajusta pools manualmente além da propriedade do Spring
