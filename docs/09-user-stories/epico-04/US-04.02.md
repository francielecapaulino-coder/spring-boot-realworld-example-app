## História de usuário

**Como** engenheiro responsável pela modernização do runtime,
**quero** que o build use o Java 25 (toolchain) e o Gradle 9.3.1 sem nenhum warning de deprecação —
**para que** o upgrade do Spring Boot (US-04.03) ocorra sobre uma base de build moderna e estável.

---

## Contexto

O projeto roda hoje sobre **Java 11** e usa um Gradle Wrapper antigo. O alvo é:

- **Java 25** (LTS, GA set/2025) via Gradle toolchain (`languageVersion = JavaLanguageVersion.of(25)`)
- **Gradle 9.3.1** via `gradle/wrapper/gradle-wrapper.properties` (`distributionUrl`)
- Build **sem warnings de deprecação** (`./gradlew build --warning-mode all` → zero `[deprecated]`)

A imagem Docker (US-03.01) usa hoje `eclipse-temurin:11`. A migração do runtime para Java 25 no `Dockerfile` é tratada nesta história (build + runtime stages) ou referenciada como ajuste acoplado, mantendo a multi-stage existente.

### Restrições de arquitetura aplicáveis

- **ADR-004:** a tarefa `generateJava` (DGS) deve continuar funcionando — não editar `io.spring.graphql.*`.
- Nenhuma mudança de contrato de API.

---

## O que esta história faz — e o que NÃO faz

### ✅ O que esta história faz

- Atualiza o Gradle Wrapper para **9.3.1** (`gradle-wrapper.properties`)
- Configura a **Java toolchain 25** no `build.gradle`
- Atualiza as imagens base do `Dockerfile` para Java 25 (build e runtime)
- Garante `./gradlew build` sem warnings de deprecação
- Ajusta plugins/tarefas incompatíveis com Gradle 9.3.1 (mínimo necessário para o build passar)

### ❌ O que esta história NÃO faz

- Não atualiza o Spring Boot — isso é US-04.03
- Não migra `javax.*` → `jakarta.*` — isso é US-04.04
- Não atualiza o DGS — isso é US-04.06
- Não introduz record types — EPIC-06
