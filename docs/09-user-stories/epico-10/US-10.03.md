## História de usuário

**Como** operador da plataforma,  
**quero** que a aplicação emita logs JSON estruturados nos eventos de startup e shutdown,  
**para que** o Loki capture esses eventos e o script `validate_startup.py` possa validá-los.

---

## Contexto

O script `scripts/validate_startup.py` já existe e valida o log de startup — verificar o formato esperado antes de implementar.

**Eventos requeridos:**

| Evento | Trigger Spring | Campos mínimos |
|---|---|---|
| Startup | `ApplicationStartedEvent` | `event: "APPLICATION_STARTED"`, `timestamp`, `version`, `port` |
| Shutdown | `ContextClosedEvent` | `event: "APPLICATION_STOPPED"`, `timestamp`, `uptime_seconds` |

---

## Escopo

* Criar `ApplicationLifecycleLogger.java` em `io.spring.infrastructure.observability` (ou pacote equivalente)
* Implementar `ApplicationListener<ApplicationStartedEvent>` para log de startup
* Implementar `ApplicationListener<ContextClosedEvent>` para log de shutdown
* Configurar Logback com `LogstashEncoder` (ou equivalente JSON) em `logback-spring.xml`
* Verificar que `scripts/validate_startup.py` passa após a mudança

## Fora de escopo

* Log de cada requisição HTTP — o Spring Boot Actuator já provê via `http.server.requests`
* Logs de auditoria de negócio

## Critérios de aceite

* [ ] Log de startup emitido como JSON com campos `event`, `timestamp`, `version`, `port`
* [ ] Log de shutdown emitido como JSON com campos `event`, `timestamp`, `uptime_seconds`
* [ ] `python3 scripts/validate_startup.py` passa sem erro
* [ ] Logs enviados para Loki quando `docker-compose up` está ativo
* [ ] `./gradlew test` 100% verde

## Regras técnicas

* Usar `logstash-logback-encoder` (biblioteca existente? verificar `build.gradle`) para JSON — se não existir, adicionar
* Não usar `System.out.println` — usar `Logger` SLF4J
* Uptime calculado como diferença entre startup time e shutdown time

## Dependências

* US-10.01 concluído (Loki configurado)

## Resultado esperado

Logs estruturados JSON de startup e shutdown capturáveis pelo Loki e validáveis pelo script Python.

## Observações para o agente

* Leia `scripts/validate_startup.py` antes de implementar — o formato do log deve ser compatível com o que o script espera.
* Verificar se `logstash-logback-encoder` já está no `build.gradle`.
* Após merge, atualizar `05-backlog.md` e fechar issue.
