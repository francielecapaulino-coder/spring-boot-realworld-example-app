## História de usuário

**Como** sistema de plataforma,  
**quero** calcular e persistir o `readingTimeMinutes` de artigos existentes na primeira vez que são lidos,  
**para que** não seja necessário processar toda a base histórica em batch e o campo fique disponível sem downtime.

---

## Contexto

US-13.01 garante que novos artigos (criados ou editados após o deploy) já têm `reading_time_minutes` calculado. Artigos históricos têm o campo `NULL`. Esta história implementa o lazy update: quando `GET /articles/:slug` é chamado e o artigo tem `reading_time_minutes = NULL`, o sistema calcula, persiste e retorna o valor — tudo na mesma requisição. Na segunda leitura, o valor já está no banco e é retornado diretamente.

**Comportamento por cenário:**

| Cenário | O que acontece |
|---|---|
| Artigo novo (pós US-13.01) | Campo já preenchido — retorno direto |
| Artigo existente — 1ª leitura | `NULL` detectado → calcula → persiste → retorna |
| Artigo existente — 2ª leitura | Valor no banco → retorno direto |
| Artigo editado (body mudou) | US-13.01 já recalculou — retorno direto |

---

## Escopo

* Em `ArticleService.findBySlug()` (ou método equivalente de leitura de artigo único): verificar se `readingTimeMinutes == null`
* Se `null`: chamar `ReadingTimeCalculator.calculate(article.getBody())`, persistir via `articleRepository.save()` e retornar valor calculado
* Garantir que a persistência lazy não quebre leituras somente-leitura (usar `@Transactional` adequado)
* Adicionar testes de integração: artigo sem campo → primeira leitura preenche; segunda leitura retorna valor persistido

## Fora de escopo

* Batch job para pré-popular todos os artigos históricos
* Cache em memória (Redis, Caffeine) — o banco é o cache; in-memory é complexidade desnecessária agora
* Recálculo automático em caso de edição (já coberto em US-13.01)

## Critérios de aceite

* [ ] `GET /articles/:slug` de artigo com `reading_time_minutes = NULL` → retorna `readingTimeMinutes` calculado e persiste no banco
* [ ] Segunda chamada ao mesmo slug → `readingTimeMinutes` retorna do banco (sem novo cálculo)
* [ ] Nenhuma leitura em listagem (`GET /articles`) dispara lazy update (apenas leitura de slug individual)
* [ ] Testes de integração cobrem os dois cenários (null → calculado; já preenchido → retorno direto)
* [ ] `./gradlew test` 100% verde

## Regras técnicas

* Lazy update apenas em `GET /articles/:slug` — não em listagens (evitar N+1 implícito)
* Transação deve ser `@Transactional` na camada de serviço para garantir atomicidade do calcular + salvar
* Não usar `@Cacheable` Spring (cache in-memory) — o banco é a fonte de verdade

## Dependências

* US-13.01 concluído (coluna `reading_time_minutes` existe, `ReadingTimeCalculator` disponível)

## Resultado esperado

Artigos históricos têm `readingTimeMinutes` preenchido on-demand, sem batch e sem impacto em listagens. O campo torna-se progressivamente populado à medida que os artigos são acessados.

## Observações para o agente

* Leia `docs/AGENTS.md` e `00-original-mandate.md` (J8) antes de implementar.
* Não adicionar Redis ou qualquer cache in-memory — fora do escopo.
* Cuidado com o contexto de transação: se o método de leitura for `@Transactional(readOnly=true)`, o save vai falhar — ajustar para `@Transactional` padrão ou criar método separado.
* Após merge, atualizar `05-backlog.md` e fechar issue correspondente.
