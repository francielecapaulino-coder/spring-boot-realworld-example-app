## História de usuário

**Como** tech lead,  
**quero** que o Pitest rode automaticamente no CI e quebre o build se o score cair abaixo de 95%,  
**para que** regressões de qualidade de teste sejam detectadas em todo PR.

---

## Contexto

Após US-07.01 a US-07.04, o score deve estar ≥ 95% localmente. Esta história garante que:
1. O CI (GitHub Actions — `.github/workflows/gradle.yml`) executa `./gradlew pitest`
2. O build quebra automaticamente se o threshold cair
3. O relatório HTML é publicado como artefato do CI

**KR3.1:** "Pitest mutation score ≥ 95%" — critério de saída do EPIC-07.

---

## Escopo

* Adicionar step `pitest` ao job do GitHub Actions em `.github/workflows/gradle.yml`
* Configurar upload do relatório HTML como artifact (`actions/upload-artifact`)
* Verificar que `mutationThreshold = 95` e `coverageThreshold = 95` no `build.gradle` (já em US-07.01)
* Executar `./gradlew pitest` localmente e confirmar score ≥ 95% antes do PR
* Documentar evidência em `docs/process/tests-epico-07/pitest-evidence.md`

## Fora de escopo

* Cache de resultados Pitest no CI (otimização futura)
* Parallelização avançada do Pitest além de `threads = 4`

## Critérios de aceite

* [ ] `.github/workflows/gradle.yml` contém step `./gradlew pitest`
* [ ] Build quebra automaticamente se score < 95%
* [ ] Relatório HTML publicado como artifact no CI
* [ ] `./gradlew pitest` local retorna score ≥ 95%
* [ ] `docs/process/tests-epico-07/pitest-evidence.md` criado com evidência

## Regras técnicas

* O step Pitest deve rodar APÓS o step de testes (`./gradlew test`)
* `excludedClasses = ['io.spring.graphql.*']` — código gerado DGS não faz parte do score (já configurado em US-07.01)

## Dependências

* US-07.01 a US-07.04 concluídos
