## Contexto técnico completo

### Por que a separação entre PARTE A e PARTE B importa

O Coda é uma ferramenta com interface gráfica — páginas, tabelas e campos são criados via UI, não via código ou API pública. Uma LLM não pode criar o workspace automaticamente. O que a LLM pode fazer é:
- Gerar o conteúdo de `docs/process/coda-guide.md`
- Atualizar o `README.md` com o link

A PARTE A (workspace Coda) é realizada **manualmente pela PM Franciele ou pelo tech lead** antes do vibe coding da PARTE B começar. A URL obtida na PARTE A é o insumo necessário para a PARTE B.

### Fluxo correto desta história

```
1. PM/Tech Lead cria workspace no Coda (PARTE A — manual)
      ↓
2. PM/Tech Lead copia a URL do workspace
      ↓
3. URL fornecida como contexto para o vibe coding
      ↓
4. LLM gera coda-guide.md e atualiza README (PARTE B — vibe coding)
      ↓
5. Commit + Push + PR no fork
```

### Por que Coda e não Notion ou Google Docs

Coda combina documentos e planilhas em uma única ferramenta. Cada história pode ter uma linha em uma tabela de rastreamento E uma página com sub-tabelas de etapas — sem alternar ferramentas. A estrutura Documento → Página por Épico → Tabela de Histórias → Sub-tabela de Etapas é nativa no Coda.

## Por que o Coda é obrigatório

A gestão exige: *"Code and commits with Coda"*.

O Coda é a memória do processo de desenvolvimento. Ele responde:
- Qual prompt gerou este código?
- Qual modelo de IA foi usado nesta etapa?
- O que precisou de ajuste manual após o output da LLM?
- Quais skills o time desenvolveu ao longo do projeto?

Sem o Coda, o desenvolvimento com IA é uma caixa preta.
Com o Coda, é um processo rastreável, reproduzível e auditável.

| Momento | Ação no Coda |

## Como documentar uma etapa de desenvolvimento

### Estrutura de cada etapa

```
Etapa:           2/5
Descrição:       Implementar calculateReadingTime no ArticleService
Prompt:          (prompt completo — ver formato abaixo)
Modelo/Tool:     Claude Sonnet 4.6
Output resumido: Método com MAX(1, CEIL(wordCount/200)). Trata body vazio.
Commit bleeding: https://github.com/francielecapaulino-coder/.../commit/abc1234
Skills:          MAX() e CEIL() em Java; tratamento de edge case (body vazio)
Ajustes:         Nenhum — output correto na primeira tentativa
Tempo:           12 min
Status:          ✅ Concluída
```

### Formato obrigatório do prompt no Coda

Documentar o prompt **completo** — não apenas o resumo.
O valor do Coda está em poder reproduzir exatamente o que foi feito.

```
Estou implementando a etapa 2/5 da história US-13.02 do projeto
RealWorld Platform Modernization.

Repositório: francielecapaulino-coder/spring-boot-realworld-example-app
Branch: feat/us-13.02-reading-time-calculation

Contexto:
- ADR-002: Spring Data JPA — estratégia de queries
- Camada: io.spring.application (casos de uso)
- Fórmula: MAX(1, CEIL(wordCount / 200))
- wordCount = body.trim().split("\\s+").length
- Retorno mínimo: 1 (nunca 0)

O que preciso:
Criar o método calculateReadingTime(String body) na ArticleService.
O método deve:
1. Tratar body null ou em branco retornando 1
2. Contar palavras separando por whitespace
3. Aplicar MAX(1, CEIL(wordCount / 200))
4. Retornar int com Javadoc explicando a fórmula

Não criar testes nesta etapa — apenas o método de produção.
```
