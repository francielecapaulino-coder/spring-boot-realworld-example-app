## Contexto

O mandato exige dois branches: um com testes falhando (`feat/playwright-broken`) e outro com testes funcionando (`feat/playwright-working`). Esta história cria o primeiro.

**Por que testes falhando?** Os fluxos E2E tentam usar funcionalidades que ainda não estão implementadas (soft delete, reading time) ou dependem de configuração que ainda será feita.

**Estratégia de falha intencional:** escrever os testes contra endpoints/comportamentos corretos, mas com assertions que sabidamente falham no estado atual do código (ex: verificar `readingTimeMinutes` antes de EPIC-13, ou verificar ausência de artigo deletado antes de EPIC-12).

---

## Escopo

* No branch `feat/playwright-broken`, escrever os 3 fluxos mínimos (MT) com assertions que falham intencionalmente:
  - **Fluxo 1 (Registro e login):** `POST /users` → `POST /users/login` → assert campo futuro
  - **Fluxo 2 (Criar artigo):** Login → `POST /articles` → `GET /articles/:slug` → assert `readingTimeMinutes` (falha pois EPIC-13 não está feito)
  - **Fluxo 3 (Soft delete):** Login → criar artigo → `DELETE /articles/:slug` → `GET /articles` → assert ausência (falha pois EPIC-12 não está feito)
* Push do branch `feat/playwright-broken` no GitHub — CI deve ser vermelho
* Documentar evidência de CI vermelho em `docs/process/tests-epico-09/broken-branch-evidence.md`
