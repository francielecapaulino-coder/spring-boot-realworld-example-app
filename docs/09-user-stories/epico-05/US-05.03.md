## História de usuário

**Como** mantenedor da camada de persistência,<br>
**quero** substituir os repositórios MyBatis de escrita por implementações Spring Data JPA,<br>
**para que** as operações de escrita usem JPA + Hibernate sem dependência de mappers XML.

---

## Contexto

Existem 4 repositórios de escrita MyBatis que implementam interfaces do `core`:

| Implementação atual | Interface do `core` | Operações |
|---|---|---|
| `MyBatisArticleRepository` | `ArticleRepository` | `save` (insert/update), `findById`, `findBySlug`, `remove` |
| `MyBatisUserRepository` | `UserRepository` | `save`, `findById`, `findByUsername`, `findByEmail`, `saveRelation`, `findRelation`, `removeRelation` |
| `MyBatisCommentRepository` | `CommentRepository` | `save`, `findById(articleId, id)`, `remove` |
| `MyBatisArticleFavoriteRepository` | `ArticleFavoriteRepository` | `save`, `find(articleId, userId)`, `remove` |

**Padrão de migração:**
1. Criar interface `JpaRepository<Entidade, String>` para cada entidade
2. Criar nova implementação (ex: `JpaArticleRepository`) que injeta o `JpaRepository` e implementa a interface do `core`
3. Manter a interface do `core` **inalterada** — contrato de domínio não muda
4. Remover o `@Repository` da implementação MyBatis (mas não deletar ainda — remoção em US-05.05)

**Lógica específica de `MyBatisArticleRepository.save`:**
Verifica se o artigo já existe (`findById`) para decidir entre insert e update. Com JPA, usar `entityManager.merge()` ou confiar no estado gerenciado (`@Transactional` + `save()` do `JpaRepository` que já faz upsert por `@Id`).

**Lógica de `UserRepository.saveRelation/findRelation/removeRelation`:**
`FollowRelation` tem chave composta — o `JpaRepository` de `FollowRelation` precisa do ID composto configurado em US-05.02.

## Escopo

* Criar `SpringDataArticleRepository extends JpaRepository<Article, String>` com `findBySlug(String slug)`
* Criar `JpaArticleRepository implements ArticleRepository` usando `SpringDataArticleRepository`
* Criar `SpringDataUserRepository extends JpaRepository<User, String>` com `findByUsername`, `findByEmail`
* Criar `SpringDataFollowRelationRepository extends JpaRepository<FollowRelation, FollowRelationId>`
* Criar `JpaUserRepository implements UserRepository` usando ambos os repositórios Spring Data acima
* Criar `SpringDataCommentRepository extends JpaRepository<Comment, String>` com `findByArticleIdAndId`
* Criar `JpaCommentRepository implements CommentRepository`
* Criar `SpringDataArticleFavoriteRepository extends JpaRepository<ArticleFavorite, ArticleFavoriteId>`
* Criar `JpaArticleFavoriteRepository implements ArticleFavoriteRepository`
* Anotar todas as implementações novas com `@Repository @Primary` para sobrepor o MyBatis enquanto ambas coexistem
* `./gradlew test` deve passar após cada implementação

## Fora de escopo

* Read services (consultas de leitura complexas) — US-05.04
* Remoção dos arquivos MyBatis — US-05.05
* Queries GraphQL/feed — US-05.04

## Critérios de aceite

* [ ] As 4 interfaces do `core` são implementadas por classes JPA (sem MyBatis)
* [ ] `./gradlew test` 100% verde após cada implementação
* [ ] Nenhum `import org.apache.ibatis.*` nos arquivos novos
* [ ] `POST /users`, `POST /articles`, `POST /articles/:slug/comments` funcionam via integração
* [ ] `DELETE /articles/:slug` e `DELETE /articles/:slug/comments/:id` funcionam

## Regras técnicas

* ADR-002: Queries por derivação de nome (`findBySlug`, `findByEmail`, `findByUsername`) — não usar JPQL para queries simples
* `@Transactional` nas implementações que fazem múltiplas operações (ex: `save` de artigo com tags)
* `@Primary` nas implementações JPA para que o Spring injete a versão correta enquanto MyBatis ainda está no classpath
* Operações de `remove`: chamar `delete()` do JPA — **exceção futura**: após EPIC-12 (soft delete), `delete` de `Article` será substituído por `is_deleted=true`; documentar esse ponto com comentário `// TODO EPIC-12: substituir por soft delete`
