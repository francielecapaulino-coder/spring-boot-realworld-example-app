## História de usuário

**Como** mantenedor da camada de consultas,  
**quero** substituir os read services MyBatis por implementações JPA/JPQL,  
**para que** todas as queries de leitura usem Hibernate sem mappers XML.

---

## Contexto

Os read services são interfaces `@Mapper` do MyBatis consumidas pelas queries de leitura. São a parte mais complexa da migração porque contêm queries com JOINs, paginação com cursor e filtros dinâmicos:

| Interface MyBatis atual | Pacote | Queries principais |
|---|---|---|
| `ArticleReadService` | `readservice` | `findById`, `findBySlug`, `queryArticles` (filtros dinâmicos: tag, author, favoritedBy + paginação offset), `findArticles`, `findArticlesOfAuthors`, `findArticlesWithCursor`, `countArticle`, `countFeedSize` |
| `UserReadService` | `readservice` | `findByUsername`, `findById` — retorna `UserData` (DTO, não entidade) |
| `CommentReadService` | `readservice` | `findById`, `findByArticleId`, `findByArticleIdWithCursor` |
| `TagReadService` | `readservice` | `all()` — lista todas as tags |
| `ArticleFavoritesReadService` | `readservice` | queries de contagem de favoritos |
| `UserRelationshipQueryService` | `readservice` | queries de following/followers |

**ADR-002 (obrigatório):** seguir a ordem: derivação → JPQL → SQL nativo. Não saltar direto para SQL nativo sem avaliar se JPQL resolve.

**Ponto mais complexo — `ArticleReadService.queryArticles`:**
Tem 3 filtros opcionais (`tag`, `author`, `favoritedBy`) + paginação offset. Com JPA, implementar via `@Query` JPQL com parâmetros opcionais, ou via `Specification` (Spring Data). Preferir `@Query` se a query for única; `Specification` se os filtros forem combinações arbitrárias.

## Escopo

* Criar `JpaArticleReadService` com métodos equivalentes aos de `ArticleReadService`, usando `@Query` JPQL ou Spring Data query derivation
* Criar `JpaUserReadService` equivalente a `UserReadService` — retorna `UserData` via JPQL projection ou `interface-based projection`
* Criar `JpaCommentReadService` equivalente a `CommentReadService`
* Criar `JpaTagReadService` equivalente a `TagReadService` — `findAll()` do `JpaRepository<Tag, String>`
* Criar `JpaArticleFavoritesReadService` equivalente a `ArticleFavoritesReadService`
* Criar `JpaUserRelationshipQueryService` equivalente a `UserRelationshipQueryService`
* Os novos serviços devem ser anotados com `@Service @Primary` para sobrepor os MyBatis
* Os tipos de retorno devem ser os mesmos (ex: `ArticleData`, `UserData`, `CommentData`) — sem alterar contratos upstream
* `./gradlew test` deve passar após cada serviço migrado

## Fora de escopo

* Alteração nos DTOs de retorno (`ArticleData`, `UserData`, `CommentData`) — contratos inalterados
* Paginação com cursor para GraphQL — manter comportamento idêntico ao MyBatis, apenas trocar a implementação
* Otimizações de performance (N+1, fetch joins) — podem ser feitas posteriormente; o objetivo aqui é paridade funcional

## Critérios de aceite

* [ ] Todos os 6 read services têm implementação JPA equivalente
* [ ] `GET /articles` com filtros `tag`, `author`, `favorited` retorna resultados corretos
* [ ] `GET /articles/feed` retorna artigos dos autores seguidos
* [ ] `GET /tags` retorna lista de tags
* [ ] `GET /articles/:slug/comments` retorna comentários
* [ ] `./gradlew test` 100% verde
* [ ] Nenhum `import org.apache.ibatis.*` nos arquivos novos

## Regras técnicas

* ADR-002: para `queryArticles` com filtros opcionais — avaliar `@Query` JPQL com `IS NULL OR` antes de usar `Specification`; usar `Specification` apenas se a query JPQL ficar ilegível
* Projections JPA: para retornar `UserData` (DTO, não entidade) usar constructor expression JPQL: `SELECT new io.spring.application.data.UserData(u.id, u.email, ...)` ou interface-based projection
* Paginação: `Page` e `CursorPageParameter` são classes da camada de aplicação — manter como parâmetros; traduzir para `Pageable` do Spring Data internamente

## Resultado esperado

Todos os read services roteados pelo JPA/JPQL. Comportamento da API idêntico ao MyBatis. Nenhum `@Mapper` do MyBatis usado na camada de leitura.

## Observações para o agente

* Leia `docs/AGENTS.md` (ADR-002) antes de escrever qualquer `@Query`.
* Migrate um serviço por vez e rode `./gradlew test` após cada um — não migre todos de uma vez.
* `ArticleReadService` é o mais complexo — deixe para o final após validar os mais simples.
* Se uma query JPQL ficar com mais de 5 condições e ilegível, documente a decisão de usar SQL nativo com `@Query(nativeQuery=true)` e um comentário explicativo.
* Após merge, atualizar `05-backlog.md` e fechar issue.
