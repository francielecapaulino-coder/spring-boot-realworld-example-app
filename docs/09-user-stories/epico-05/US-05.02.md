## História de usuário

**Como** mantenedor da camada de domínio,<br>
**quero** anotar as entidades existentes com `@Entity` e mapeamentos JPA,<br>
**para que** o Hibernate reconheça o modelo de dados sem alterar o schema do banco.

---

## Contexto

As classes de domínio (`Article`, `User`, `Comment`, `Tag`, `ArticleFavorite`, `FollowRelation`) são POJOs sem anotações JPA. O schema já existe via Flyway (`V1__create_tables.sql`). Esta história adiciona as anotações JPA sem alterar a lógica de negócio nem o schema.

**Mapeamento necessário:**

| Classe | Tabela (Flyway) | Observações |
|---|---|---|
| `Article` (`core/article`) | `articles` | `id` String (UUID), `slug`, `title`, `description`, `body`, `userId`, `createdAt`, `updatedAt`; tem `List<Tag>` — relação `@ManyToMany` via tabela `article_tags` |
| `User` (`core/user`) | `users` | `id` String (UUID); campo `password` com `@Column(name="password")`; sem relação direta com `FollowRelation` |
| `Comment` (`core/comment`) | `comments` | `id` String (UUID); `articleId` e `userId` como `String` FK (não objetos — manter assim para não alterar lógica de negócio) |
| `Tag` (`core/article`) | `tags` | `id` String (UUID), `name`; lado inverso da relação `@ManyToMany` com `Article` |
| `ArticleFavorite` (`core/favorite`) | `article_favorites` | Chave composta (`articleId`, `userId`); usar `@IdClass` ou `@EmbeddedId` |
| `FollowRelation` (`core/user`) | `follow_relations` | 2 campos String (`userId`, `targetId`); chave composta — usar `@IdClass` ou `@EmbeddedId` |

**Restrições críticas:**
- `Article.update()` muta campos via setters privados — o Hibernate precisa de acesso via reflexão (`@Access(AccessType.FIELD)`) ou os campos devem ser `package-private`. **Não alterar a lógica de `update()`.**
- `User.update()` tem mesma situação.
- `spring.jpa.hibernate.ddl-auto=validate` (configurado em US-05.01) — o Hibernate vai validar o schema. Se a anotação não bater com `V1__create_tables.sql`, o startup falha. Verificar nomes de colunas no SQL antes de anotar.

---

## Escopo

* Anotar `Article` com `@Entity @Table(name="articles")` e `@Id @Column` em cada campo
* Mapear `List<Tag>` em `Article` com `@ManyToMany @JoinTable(name="article_tags", ...)`
* Adicionar `@NoArgsConstructor` (Lombok) onde necessário para Hibernate (requer construtor sem args)
* Anotar `User` com `@Entity @Table(name="users")`
* Anotar `Comment` com `@Entity @Table(name="comments")`
* Anotar `Tag` com `@Entity @Table(name="tags")` e lado inverso `@ManyToMany(mappedBy="tags")`
* Anotar `ArticleFavorite` com `@Entity @Table(name="article_favorites")` e chave composta
* Anotar `FollowRelation` com `@Entity @Table(name="follow_relations")` e chave composta
* Verificar `V1__create_tables.sql` para confirmar nomes exatos de tabelas e colunas antes de cada anotação
* Confirmar startup: `./gradlew bootRun` deve passar a validação DDL do Hibernate sem erros

## Fora de escopo

## Critérios de aceite

* [ ] Todas as 6 classes têm `@Entity` e compilam
* [ ] `./gradlew bootRun` inicia sem erro de validação DDL (`spring.jpa.hibernate.ddl-auto=validate`)
* [ ] Nenhum campo `javax.*` introduzido — apenas `jakarta.persistence.*`
* [ ] `./gradlew test` 100% verde (MyBatis ainda ativo nesta história)
* [ ] `V1__create_tables.sql` não foi alterado

## Regras técnicas

* Usar `jakarta.persistence.*` — nunca `javax.persistence.*` (ADR EPIC-04)
* `@Access(AccessType.FIELD)` nas entidades com campos privados mutáveis (`Article`, `User`) para o Hibernate acessar via reflexão sem precisar de setters públicos
* Para chaves compostas (`ArticleFavorite`, `FollowRelation`): criar `@Embeddable` ID class ou usar `@IdClass` — preferir `@EmbeddedId` (mais explícito)
* `ddl-auto=validate` é o gatekeeping: se a anotação diverge do schema real, o startup falha imediatamente — isso é intencional

## Dependências

* US-05.01 concluído (`spring-boot-starter-data-jpa` no classpath, `ddl-auto=validate` configurado)
* `V1__create_tables.sql` disponível para consulta de nomes de tabelas/colunas

## Resultado esperado
