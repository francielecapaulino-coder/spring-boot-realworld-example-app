## Contexto tГ©cnico completo

### Como o Spring Boot resolve variГЎveis de ambiente

O Spring Boot suporta dois formatos de placeholder em `application.properties`:

**Sem valor padrГЈo вЂ” fail-fast:**
````properties
jwt.secret=${JWT_SECRET}
````
Se `JWT_SECRET` nГЈo estiver definida como variГЎvel de ambiente, o Spring Boot
lanГ§a `IllegalArgumentException: Could not resolve placeholder 'JWT_SECRET'`
na inicializaГ§ГЈo e a aplicaГ§ГЈo **nГЈo sobe**. Este Г© o comportamento correto
para segredos criptogrГЎficos вЂ” nГЈo existe valor padrГЈo seguro.

**Com valor padrГЈo вЂ” fallback:**
````properties
jwt.sessionTime=${JWT_SESSION_TIME:86400}
````
Se `JWT_SESSION_TIME` nГЈo estiver definida, o Spring Boot usa `86400` como
fallback. A aplicaГ§ГЈo sobe normalmente. Adequado para configuraГ§Гµes operacionais
que tГЄm um valor razoГЎvel como padrГЈo.

### DecisГЈo arquitetural de referГЄncia

> **ADR-006** (`06-architecture-decisions.md` v1.0) define:
> - `JWT_SECRET`: sem fallback вЂ” fail-fast. Chave criptogrГЎfica nГЈo pode ter valor padrГЈo seguro.
> - `JWT_SESSION_TIME`: fallback `86400` (24h). ConfiguraГ§ГЈo operacional com padrГЈo razoГЎvel.

### Por que nГЈo alterar `DefaultJwtService.java` nesta histГіria

O `DefaultJwtService.java` usa `javax.crypto.SecretKey` вЂ” um import `javax.*`
que serГЎ migrado para `jakarta.*` em EPIC-04 (US-04.04). Alterar o arquivo
agora criaria um conflito com a migraГ§ГЈo futura. O escopo desta histГіria Г©
exclusivamente o `application.properties`.
