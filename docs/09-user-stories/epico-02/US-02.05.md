## História de usuário

**Como** time de desenvolvimento que recebe código gerado por vibe coding de múltiplas duplas,
**quero** que todo push e toda pull request sejam escaneados automaticamente em busca de secrets antes de qualquer build ou teste rodar,
**para que** nenhum valor sensível (chave JWT, senha de banco, token de API) chegue ao histórico do Git por engano, mesmo que um desenvolvedor esqueça de revisar manualmente.

---

## Problema atual

Hoje o pipeline de CI (`.github/workflows/gradle.yml`) executa apenas build e testes. Não existe nenhuma verificação de que o código contém ou não valores sensíveis hardcoded. Isso significa que:

- Um futuro `jwt.secret=alguma-coisa-fixa` (regressão de US-02.01) passaria pelo CI sem ser detectado
- Uma nova variável de ambiente introduzida por engano com um valor real (em vez de placeholder) em `.env.example` não seria pega
- A única defesa hoje é revisão humana de PR — que falha exatamente nos momentos de maior pressa, quando o risco é maior

O `.env.example` atual já antecipa esta história em comentário (`# Protecao complementar: US-02.05 configura truffleHog no CI...`), mas a proteção real ainda não existe.

## Decisão — TruffleHog vs git-secrets

A Definition of Ready de INI-02 (`03-initiatives.md`) exige explicitamente: *"Escolha feita entre truffleHog e git-secrets"* antes de iniciar. Esta história formaliza essa escolha.

| Critério | TruffleHog | git-secrets |
|---|---|---|
| Verificação ativa de credenciais (não é só regex) | ✅ Sim — testa a credencial contra a API real do provedor (`verified` / `unverified` / `unknown`) | ❌ Não — apenas casamento de padrão |
| Falsos positivos | Baixo — resultado `verified` confirma que a credencial é real | Alto — qualquer string parecida com chave dispara alerta |
| Suporte oficial como GitHub Action | ✅ `trufflesecurity/trufflehog` mantido ativamente (release mais recente: `v3.95.5`) | ⚠️ Ferramenta da AWS Labs, sem Action oficial mantida da mesma forma |
| Já referenciado no projeto | ✅ Citado em `.env.example`, `M-SEC-01` (`07-metrics.md`), critério de aceitação do EPIC-02 | ❌ Não citado em nenhum documento |
| Scan de histórico completo do Git | ✅ Sim | ✅ Sim (hooks de pre-commit + scan de histórico) |

**Decisão: TruffleHog OSS**, via a GitHub Action oficial `trufflesecurity/trufflehog`. Critério decisivo: verificação ativa reduz o ruído de falsos positivos que tornaria o gate do CI irritante o suficiente para o time querer contorná-lo — e é a ferramenta que toda a documentação já assumia implicitamente.

## O que esta história entrega

- Um novo job `secret-scan` em `.github/workflows/gradle.yml`, executado **antes** do job `build` (via `needs: secret-scan`) — cumprindo a regra de negócio do EPIC-02: *"Scan no CI é o primeiro step: executa antes do build e dos testes"*
- O job falha (e bloqueia o `build`) se o TruffleHog encontrar qualquer credencial `verified` ou `unknown` no diff escaneado
- Atualização do comentário em `.env.example` que hoje promete esta história como trabalho futuro
- Validação manual: um commit com um secret simulado em uma branch de teste faz o CI falhar de forma clara

## O que esta história NÃO faz

| O que | Onde será feito |
|---|---|
| Scan de secrets fora do GitHub (ex: Slack, Confluence) | Fora de escopo — TruffleHog Enterprise, não OSS |
| Rotação de secrets já expostos no histórico antigo | Fora de escopo do projeto (ver `03-initiatives.md`, INI-02, "fora do escopo") |
| Limpeza retroativa do histórico Git | Fora de escopo — risco maior que o benefício (`03-initiatives.md`) |
| Documentar configuração de ambiente no `CONTRIBUTING.md` | US-02.06 |
| Bloquear push direto na `master` | Já é regra do AGENTS.md (lista vermelha) — não depende de scan de secrets |

## Contexto técnico completo

### Onde o scan se encaixa no pipeline atual

```
Push ou PR
    ↓
[NOVO] Job secret-scan ─── falha? ──→ pipeline para aqui, build nunca roda
    ↓ (passou)
Job build (needs: secret-scan)
    ↓
Checkout → JDK 11 → cache Gradle → JWT_SECRET de teste → ./gradlew clean test
```

`needs: secret-scan` é o mecanismo do GitHub Actions que garante a ordem — os dois jobs do mesmo workflow (`gradle.yml`) agora têm dependência explícita. Não é possível usar `needs` entre arquivos de workflow diferentes; por isso o novo job entra no mesmo arquivo do job `build`, não em um workflow separado.

### Como a TruffleHog Action funciona neste contexto

A action faz checkout do histórico completo (`fetch-depth: 0`) e compara o `base` (branch padrão do repositório) com o `head` (commit atual). Ela escaneia apenas o diff entre os dois — não o repositório inteiro a cada execução, o que mantém o CI rápido. Quando uma credencial com status `verified` ou `unknown` é encontrada no diff, o step da action termina com falha, e por consequência o job e o pipeline inteiro falham.

### Por que `extra_args: --results=verified,unknown` e não o padrão

| Configuração | Comportamento |
|---|---|
| Padrão (sem `extra_args`) | Reporta também resultados `unverified` — alto volume de ruído |
| `--results=verified,unknown` | Reporta apenas o que foi confirmado como credencial real (`verified`) ou que não pôde ser verificado por erro de rede/API (`unknown`, tratado como risco) — ignora `unverified` |

### Arquivo a alterar

```
.github/
└── workflows/
    └── gradle.yml          ← MODIFICADO (novo job secret-scan + needs no job build)
.env.example                ← MODIFICADO (comentário de US-02.05 atualizado para refletir conclusão)
```

Nenhum arquivo Java é tocado nesta história — é puramente configuração de CI.


---

### Passo 2 — Modificar `.github/workflows/gradle.yml`

Adicionar o job `secret-scan` antes do job `build`, e fazer `build` depender dele:

```yaml
name: Java CI

on:
  push:
    branches:
      - '**'
    tags:
      - 'v*'
  pull_request:
    branches:
      - '**'

jobs:
  secret-scan:
    name: Scan for secrets (TruffleHog)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: TruffleHog OSS
        uses: trufflesecurity/trufflehog@v3.95.5
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}
          head: HEAD
          extra_args: --results=verified,unknown

  build:
    name: Build and test
    needs: secret-scan
    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v4
    - name: Set up JDK 11
      uses: actions/setup-java@v4
      with:
        distribution: zulu
        java-version: '11'
    - uses: actions/cache@v4
      with:
        path: |
          ~/.gradle/caches
          ~/.gradle/wrapper
        key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*') }}
        restore-keys: |
          ${{ runner.os }}-gradle-
    - name: Generate JWT secret for tests
      run: echo "JWT_SECRET=$(openssl rand -base64 64 | tr -d '\n')" >> $GITHUB_ENV
    - name: Test with Gradle
      run: ./gradlew clean test
```

> **Nota:** quando `base` e `head` apontam para o mesmo commit (caso raro, ex. primeiro push de uma branch nova sem diff), a action emite um erro porque não há nada para comparar. Isso é esperado e documentado pela própria TruffleHog — não é uma falha desta implementação.

### Passo 5 — Teste de regressão simulado (validação manual obrigatória)

Em uma branch descartável (não a de trabalho), confirmar que o gate funciona:

```bash
git checkout -b test/secret-scan-validation
echo 'aws_access_key_id=AKIAIOSFODNN7EXAMPLE' >> /tmp/fake-secret-test.txt
cp /tmp/fake-secret-test.txt ./fake-secret-test.txt
git add fake-secret-test.txt
git commit -m "test: trigger secret scan validation (not for merge)"
git push origin test/secret-scan-validation
```

Esperado: o job `secret-scan` falha no GitHub Actions para esta branch. Depois de confirmar, deletar a branch e o arquivo:

```bash
git push origin --delete test/secret-scan-validation
git checkout ci/us-02.05-secret-scan-trufflehog
```

> Use a chave de exemplo pública e documentada da AWS (`AKIAIOSFODNN7EXAMPLE`) — ela é reconhecida pelos detectores como formato válido de chave AWS, mas é uma chave de documentação oficial, nunca uma credencial real.
