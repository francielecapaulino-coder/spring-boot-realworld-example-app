## História de usuário

**Como** operador da plataforma,  
**quero** que a exclusão de um artigo marque o registro como `is_deleted = true` em vez de removê-lo permanentemente,  
**para que** o conteúdo possa ser auditado, recuperado em caso de erro e o comportamento visível para o usuário permaneça idêntico.

---

## Contexto

Atualmente `DELETE /articles/:slug` executa `DELETE FROM articles WHERE slug = ?` — exclusão permanente e irreversível. EPIC-05 migrou o ORM para Spring Data JPA + Hibernate, habilitando o uso de `@Where(clause = "is_deleted = false")` na entidade para filtrar registros deletados automaticamente em todas as queries. Esta história adiciona a coluna `is_deleted` via Flyway migration, anota a entidade `Article` com `@Where` e muda o serviço de deleção para setar a flag em vez de remover.

---

## Escopo

* Criar migration Flyway: `ALTER TABLE articles ADD COLUMN is_deleted BOOLEAN NOT NULL DEFAULT false`
* Anotar entidade `Article` com `@SQLRestriction("is_deleted = false")` (Jakarta/Spring Data JPA 3+) ou `@Where` equivalente
* Modificar `ArticleRepository.delete()` para executar `UPDATE SET is_deleted = true` (via método customizado ou `@Modifying @Query`)
* Garantir que `GET /articles`, `GET /articles/:slug`, feed e favoritos **não retornam** artigos com `is_deleted = true`
* Garantir que `DELETE /articles/:slug` retorna `204 No Content` (comportamento externo inalterado)
* Adicionar testes de integração: deletar artigo → confirmar que não aparece em listagens; confirmar que slug retorna 404

## Fora de escopo

* Soft delete de comentários (US-12.02)
* Endpoint de restauração/admin de artigos deletados
* Paginação ou listagem de artigos deletados para operadores
* Mudança no comportamento de autorização do DELETE

## Critérios de aceite

* [ ] Migration Flyway executada sem erro: coluna `is_deleted` existe com default `false`
* [ ] `DELETE /articles/:slug` retorna `204` e o artigo não aparece mais em `GET /articles`
* [ ] `GET /articles/:slug` do artigo deletado retorna `404`
* [ ] Artigo deletado não aparece no feed do usuário
* [ ] Artigo deletado não aparece em listagens de favoritos
* [ ] Testes de integração cobrem: delete + listagem, delete + get por slug
* [ ] `./gradlew test` 100% verde após a mudança

## Regras técnicas

* Usar `@SQLRestriction` (Hibernate 6.x, Jakarta-based) em vez do legado `@Where` se disponível
* Migration Flyway deve ser versionada como `V<n>__add_is_deleted_to_articles.sql`
* O campo `is_deleted` **nunca** deve ser exposto no JSON de resposta da API
* Soft delete deve ser transparente para o consumidor — sem mudança no contrato externo

## Dependências

* EPIC-05 concluído (Spring Data JPA + Hibernate com suporte a `@SQLRestriction`)
* EPIC-07 + EPIC-08 concluídos (mutation testing ≥ 95% e testes de integração — condição de entrada da Fase 5)

## Resultado esperado

Artigos excluídos permanecem no banco com `is_deleted = true`, invisíveis para todos os endpoints públicos. Comportamento externo da API idêntico ao anterior. Auditoria habilitada para operações futuras.
