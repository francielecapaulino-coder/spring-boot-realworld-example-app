## História de usuário

**Como** time de qualidade do contrato de API,  
**quero** testes de integração que chamam endpoints reais do REST via @SpringBootTest ou @WebMvcTest,  
**para que** saibamos se o contrato público está funcional antes/after upgrades e migrações.

---

## Contexto

O projeto já tem alguns testes REST com REST Assured. Precisamos garantir que todos os endpoints públicos do contrato API realworld sejam cobertos: usuários, artigos, perfis, favoritos, comentários, tags. Os testes devem rodar contra H2 ou Postgres de integração (já em EPIC-03), gerar tokens JWT para autenticação, e validar status code e payload. Não é necessário 100% de casos de borda, mas sim o fluxo principal e edgepaths básicos.

---

## Escopo

* Escrever `UserApiIntegrationTest` com signup e login
* Escrever `ArticleApiIntegrationTest` com CRUD de artigo (com autenticação)
* Escrever `ProfileApiIntegrationTest` com follow/unfollow
* Escrever `FavoriteApiIntegrationTest` com favorite/unfavorite
* Escrever `CommentApiIntegrationTest` com add/delete comment
* Escrever `TagApiIntegrationTest` para `/tags`
* Usar @SpringBootTest (ou @WebMvcTest se mais leve)
* Usar JWT real gerado em memória (TestHelper já pode gerar)
* Configurar banco H2 ou Postgres in-memory para isolamento

## Fora de escopo

* GraphQL integration tests (será parte de EPIC-XX GraphQL)
* Testes de negocio ligado a repositórios MyBatis (já existe)
* Mockar camadas externas; estes testes devem chamar controllers real

## Critérios de aceite

* [ ] `./gradlew test` executa todos os testes de integração e passam
* [ ] Todos os endpoints públicos são chamados ao menos uma vez com auth
* [ ] Tests validam status code 200, 201, 401/403 conforme esperado
* [ ] Relatório de cobertura mostra endpoints cobertos (opcional)

## Regras técnicas

* Usar SpringBootTest com @ActiveProfiles("test") com H2 inicializado de script
* Reutilizar beans de serviço da camada application; não mockar repository por enquanto
* Manter limpeza de dados entre testes usando @Transactional ou @DirtiesContext

## Dependências

* EPIC-04 (runtime moderno)
* EPIC-03 (ambientais de testes com Postgres/H2)

## Resultado esperado

Conjunto de testes de integração REST que cobrem o contrato público, servindo de regressão automática para futuras migrações.
