# US-01.02 — Criar templates de GitHub Issues por tipo de trabalho

## Identificação

| Campo | Valor |
|---|---|
| **ID** | US-01.02 |
| **Épico** | EPIC-01 — Fundação do processo de desenvolvimento |
| **Iniciativa** | INI-01 |
| **Fase** | Fase 0 |
| **Tipo** | `chore` |
| **Prioridade** | 🔴 Alta — nenhum trabalho começa sem issue criada |
| **Depende de** | US-01.01 (commitlint configurado — commits desta história já seguem o padrão) |
| **Bloqueia** | US-01.03 (template de PR), US-01.04 (DoR standalone), US-01.05 (DoD standalone), e todas as histórias das fases seguintes |

---

## Premissas da gestão atendidas por esta história

| Premissa | Como esta história atende |
|---|---|
| *"Tracking of issues using github issues, templates integrated with repositories"* | 7 templates YAML integrados ao repositório via `.github/ISSUE_TEMPLATE/` |
| *"Definition of ready and definition of done documented per repo"* | Cada template referencia `docs/process/definition-of-ready.md` e `docs/process/definition-of-done.md` como fonte de verdade + checklist resumido |
| *"Create issues before working, reference changes to repo to issues"* | `blank_issues_enabled: false` impede issues sem template; campo "PR relacionada" e orientação sobre `closes #XX` nos commits |
| *"Add Pitest or similar mutation test, cover to 95%"* | DoD dos templates `feat` e `fix` inclui "Pitest ≥ 95% mantido" para histórias com mudança de código |
| *"User Conventional Commits for all commits"* | Título de cada template segue o formato `tipo(escopo): descrição` |
| *"Code and commits with Coda"* | Campo "Prompts e skills (Coda)" obrigatório em todos os templates |
| *"Add integration tests"* | DoD dos templates `feat` e `fix` inclui "testes de integração quando aplicável" |
| *"Test with Playwright"* | Template `test.yml` criado especificamente para histórias de teste |
| *"Document current API contract with OpenAPI/Swagger"* | Template `docs.yml` criado para histórias de documentação |

---

## História de usuário

**Como** membro do time de desenvolvimento,  
**quero** que ao criar uma issue no GitHub eu encontre templates estruturados por tipo de trabalho — com DoR, DoD, rastreabilidade para épicos/ADRs e campo para Coda obrigatório —  
**para que** nenhuma linha de código seja gerada sem contexto documentado, sem critério de conclusão e sem rastreabilidade da issue ao commit.

---

## O que precisa existir ao final desta história

Ao concluir, o repositório terá **7 templates de issue** + 1 arquivo de configuração:

| Arquivo | Tipo de trabalho | Quando usar |
|---|---|---|
| `config.yml` | Configuração | Desabilita issues em branco |
| `feat.yml` | Nova funcionalidade (`feat`) | US com mudança de comportamento visível |
| `fix.yml` | Correção de bug (`fix`) | Correção de comportamento incorreto |
| `chore.yml` | Setup, dependências, configuração (`chore`) | Inclui `ci`, `perf`, `style` ¹ |
| `refactor.yml` | Refatoração (`refactor`) | Reestruturação sem mudança de comportamento |
| `test.yml` | Testes (`test`) | Integração, mutação, E2E, Playwright |
| `docs.yml` | Documentação (`docs`) | ADRs, OpenAPI, CONTRIBUTING, README |
| `spike.yml` | Investigação técnica (`spike`) | Pesquisa com timebox definido |

> ¹ **Mapeamento de tipos sem template próprio:** histórias do tipo `ci` (ex: US-01.01), `perf` (ex: US-04.08) e `style` usam o template `chore.yml`. O tipo do Conventional Commit no título é o correto (`ci(...)`, `perf(...)`, `style`), mas o template de issue é `chore.yml`.

## Contexto técnico completo

### Repositório alvo

- **URL:** `https://github.com/gothinkster/spring-boot-realworld-example-app`
- **Branch principal:** `master`
- **CI existente:** GitHub Actions com `Java CI` e `Commitlint` (configurado em US-01.01)
- **Templates de issue anteriores:** nenhum — todas as issues eram em branco

### Decisão sobre DoR/DoD nos templates

Os templates incluem um **checklist resumido** de DoR e DoD como lembrete dentro da issue. Porém, a **fonte de verdade** para Definition of Ready e Definition of Done é:
- `docs/process/definition-of-ready.md` (criado em US-01.04)
- `docs/process/definition-of-done.md` (criado em US-01.05)

Cada template referencia esses arquivos explicitamente. Se o checklist do template divergir dos docs standalone, os docs standalone prevalecem.

### Decisão sobre referência de issues nos commits

Todo commit que implementa uma issue deve referenciar a issue no corpo ou no footer do commit:

```bash
# Fecha a issue automaticamente ao mergear a PR
git commit -m "feat(articles): add reading time estimation

closes #42"

# Referencia sem fechar (para commits parciais)
git commit -m "feat(articles): add reading time calculation method

refs #42"
```

```bash
mkdir -p .github/ISSUE_TEMPLATE
```

### Passo 2 — Criar `.github/ISSUE_TEMPLATE/config.yml`

Desabilita issues em branco e inclui link para documentação:

```yaml
# .github/ISSUE_TEMPLATE/config.yml
blank_issues_enabled: false
contact_links:
  - name: "📋 Documentação do projeto"
    url: https://github.com/gothinkster/spring-boot-realworld-example-app/tree/master/docs
    about: "Consulte as iniciativas, épicos e ADRs antes de abrir uma issue"
  - name: "📐 Mapeamento de tipos para templates"
    url: https://github.com/gothinkster/spring-boot-realworld-example-app/blob/master/CONTRIBUTING.md
    about: "Tipos ci, perf e style usam o template Chore. Consulte o CONTRIBUTING.md"
```
