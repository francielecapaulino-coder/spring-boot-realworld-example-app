## Premissas da gestão atendidas por esta história

| Premissa | Como esta história atende |
|---|---|
| *"Issues criadas ANTES de codar — PR sem issue = PR rejeitada"* | Seção "Issue relacionada" com `closes #` obrigatório no topo do template |
| *"Conventional Commits em 100% dos commits"* | Seção "Tipo de mudança" com os 9 tipos do Conventional Commits |
| *"Pitest ≥ 95% como gate de qualidade"* | Item de DoD: `./gradlew pitest` ≥ 95% quando há mudança de código Java |
| *"Testes de integração com Testcontainers PostgreSQL (sem H2)"* | Item de DoD: Testcontainers com PostgreSQL real — nenhum teste usa H2/SQLite |
| *"Playwright para E2E via APIRequestContext"* | Item de DoD: fluxo E2E no Playwright atualizado quando há mudança visível |
| *"OpenAPI/Swagger para documentação de contratos"* | Item de DoD: OpenAPI/Swagger atualizado quando há mudança de contrato REST |
| *"Todo desenvolvimento documentado no Coda"* | Seção "Link do Coda" obrigatória |
| *"DoR e DoD documentados em docs/process/ como fonte de verdade"* | Comentário HTML aponta para `docs/process/definition-of-done.md` |

---

## História de usuário

**Como** revisor de Pull Requests do time,
**quero** que toda PR aberta apresente automaticamente um checklist estruturado com link para issue, tipo de mudança, ADRs consultados, DoD e link para o Coda,
**para que** nenhuma PR seja mergeada sem rastreabilidade, sem critérios de qualidade verificados e sem documentação do processo.

---

## O que precisa existir ao final desta história

Ao concluir, o repositório terá **1 arquivo**:

| Arquivo | Tipo | Função |
|---|---|---|
| `.github/PULL_REQUEST_TEMPLATE.md` | Markdown | Template exibido automaticamente ao abrir qualquer PR |

> **Importante:** PR templates no GitHub são **Markdown puro** — não suportam YAML (diferente dos Issue Forms da US-01.02).

### Decisão sobre itens condicionais

Itens marcados com **"(quando aplicável)"** não obrigam PRs de `docs` ou `chore` que não tocam código Java. Pitest, testes de integração, Playwright e OpenAPI só se aplicam quando há mudança de código ou contrato.

## Estrutura do template (8 seções)

1. **Issue relacionada** — `closes #` pré-preenchido + comentário HTML explicando obrigatoriedade e formatos (`closes #XX` / `refs #XX`)
2. **Tipo de mudança** — 9 checkboxes (feat, fix, test, docs, chore, refactor, ci, perf, style)
3. **O que foi feito** — campo livre + instrução (2-5 frases, referenciar épico e história)
4. **ADRs consultados** — 6 ADRs reais + "Nenhum ADR aplicável a esta PR"
5. **Checklist de Definition of Done** — 4 sub-seções:
   - Código e funcionalidade (3 itens)
   - Testes (6 itens)
   - Contrato e documentação (3 itens)
   - Processo e rastreabilidade (4 itens)
6. **Link do Coda** — campo obrigatório com comentário
7. **Evidência de testes** — 3 `<details>` colapsáveis (test, pitest, Playwright)
8. **Observações para o reviewer** — campo livre

## Critérios de aceitação

### CA-01 — Template aparece automaticamente
Ao abrir uma PR no GitHub, o corpo é pré-preenchido com o conteúdo de `.github/PULL_REQUEST_TEMPLATE.md`.

### CA-02 — Seção de issue obrigatória presente
O template inicia com `closes #` e comentário HTML explicando que PR sem issue não é aprovada.

### CA-03 — 9 tipos de mudança
A seção "Tipo de mudança" contém exatamente os 9 tipos do Conventional Commits como checkboxes.

### CA-04 — 6 ADRs reais listados
A seção "ADRs consultados" lista os 6 ADRs do projeto + opção "Nenhum aplicável".

### CA-05 — Checklist de DoD completo
As 4 sub-seções somam 16 itens cobrindo código, testes, contrato e processo.

### CA-06 — Item de Pitest presente
O DoD inclui: `./gradlew pitest` — mutation score ≥ 95% mantido (quando há mudança de código Java).

### CA-07 — Item de Testcontainers presente
O DoD inclui: Testcontainers com PostgreSQL real — nenhum teste usa H2 ou SQLite.

### CA-08 — Link do Coda obrigatório
A seção "Link do Coda" existe com comentário marcando como obrigatório.

### CA-09 — Seções colapsáveis de evidência
3 blocos `<details><summary>` para test, pitest e Playwright.

### CA-10 — Comentários HTML invisíveis no render
Instruções usam `<!-- -->` e não aparecem no render do GitHub.
