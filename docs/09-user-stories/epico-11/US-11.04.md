## História de usuário

**Como** desenvolvedor consumidor da API,  
**quero** ver os schemas de request e response com exemplos no Swagger UI,  
**para que** eu possa construir integrações sem precisar inspecionar o código fonte.

---

## Contexto

O springdoc gera schemas automaticamente a partir dos tipos Java, mas sem `@Schema` os campos ficam sem descrição nem exemplo. Esta história adiciona essas anotações nos DTOs de request/response mais usados.

**DTOs prioritários para anotação (`io.spring.api.*`):**
- Request: `NewUserParam`, `UpdateUserParam`, `NewArticleParam`, `UpdateArticleParam`, `NewCommentParam`
- Response: `UserData`, `ArticleData`, `CommentData`, `ProfileData`

---

## Escopo

* Adicionar `@Schema(description = "...", example = "...")` nos campos dos DTOs prioritários
* Adicionar `@Content(schema = @Schema(implementation = ...))` nas `@ApiResponse` de sucesso onde aplicável
* Verificar que o Swagger UI exibe exemplos reais nos schemas (ex: `"email": "user@example.com"`)
* Gerar arquivo `openapi.json` estático via task Gradle ou endpoint `/v3/api-docs` e commitar no repo

## Fora de escopo

* Schemas de erros de validação (422) com detalhe de campo — complexidade desnecessária agora
* Documentação de autenticação JWT no SecurityScheme — pode ser adicionado depois

## Critérios de aceite

* [ ] DTOs prioritários têm `@Schema` com `description` e `example` nos campos
* [ ] Swagger UI exibe exemplos nos schemas de request/response
* [ ] `openapi.json` ou `/v3/api-docs` acessível e válido (testável em editor.swagger.io)
* [ ] `./gradlew test` 100% verde

## Regras técnicas

* Usar `io.swagger.v3.oas.annotations.media.Schema` — não Springfox
* Não alterar lógica de negócio — somente anotações nos DTOs
