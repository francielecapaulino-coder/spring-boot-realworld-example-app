## História de usuário

**Como** operador de infraestrutura que monitora a saúde da aplicação,
**quero** que o Spring Actuator exponha os endpoints `/actuator/health`, `/actuator/info` e `/actuator/metrics` —
**para que** o Docker Compose saiba quando a aplicação está pronta para receber tráfego e o time tenha visibilidade sobre o estado do sistema sem precisar acessar logs.

---

## Contexto

O Spring Boot Actuator já está disponível no classpath do projeto (`spring-boot-actuator` é dependência transitiva do Spring Boot). O que falta é a configuração explícita que expõe os endpoints necessários e os protege adequadamente por ambiente.

A regra do épico define: **Actuator obrigatório antes do Docker Compose** — `/actuator/health` é o único meio pelo qual o Docker Compose sabe quando a aplicação está pronta (via `healthcheck`).

### Endpoints a expor

| Endpoint | Propósito | Exposição |
|---|---|---|
| `/actuator/health` | Healthcheck do Docker Compose e monitoramento | Todos os ambientes |
| `/actuator/info` | Metadados da aplicação (versão, build) | Todos os ambientes |
| `/actuator/metrics` | Métricas para Prometheus (EPIC-10) | Todos os ambientes |
| `/actuator/prometheus` | Formato Prometheus (usado pelo US-03.04) | Todos os ambientes |

---

## O que esta história faz — e o que NÃO faz

### ✅ O que esta história faz

- Adiciona dependência `micrometer-registry-prometheus` ao `build.gradle` (necessária para `/actuator/prometheus`)
- Configura `management.endpoints.web.exposure.include` em `application.properties`
- Configura `management.endpoint.health.show-details=always`
- Adiciona informações de build ao `application.properties` (`info.app.*`)
- Garante que os endpoints respondem antes e depois do Docker Compose

### ❌ O que esta história NÃO faz

- Não configura dashboards Grafana — isso é EPIC-10
- Não configura segurança por perfil nos endpoints Actuator — isso é EPIC-11
- Não instala Prometheus ou Grafana — isso é US-03.04
- Não cria alertas — isso é EPIC-10
