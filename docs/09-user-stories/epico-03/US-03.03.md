## História de usuário

**Como** desenvolvedor que precisa de um banco de dados confiável para múltiplas conexões simultâneas,
**quero** que o SQLite seja substituído por PostgreSQL 16 com migrations gerenciadas pelo Flyway —
**para que** a aplicação suporte múltiplos usuários concorrentes, as migrations sejam versionadas e reproduzíveis, e o banco de produção seja idêntico ao ambiente local.

---

## Contexto

O banco de dados atual é SQLite — um arquivo local (`dev.db`) que não suporta múltiplas conexões de escrita simultâneas. Isso torna o sistema incapaz de operar com mais de um usuário ativo ao mesmo tempo e é incompatível com o modelo de containerização.

### Regras do épico

- **SQLite proibido após este épico:** nenhum perfil pode usar SQLite (arquivo `application-test.properties` usa `jdbc:sqlite::memory:` — precisa ser migrado para H2 in-memory ou Testcontainers PostgreSQL)
- **Flyway obrigatório:** todas as migrations versionadas, aplicadas automaticamente no startup

### Estrutura de migrations Flyway

```
src/main/resources/db/migration/
├── V1__initial_schema.sql     ← schema inicial equivalente ao SQLite atual
├── V2__add_indexes.sql        ← índices para performance
```

### Configuração por perfil

| Perfil | Driver | URL |
|---|---|---|
| `dev` (local) | `postgresql` | `jdbc:postgresql://localhost:5432/realworld` |
| `dev` (Docker) | `postgresql` | `jdbc:postgresql://postgres:5432/realworld` |
| `test` | H2 in-memory | `jdbc:h2:mem:testdb;MODE=PostgreSQL` |

---

## O que esta história faz — e o que NÃO faz

### ✅ O que esta história faz

- Remove dependência SQLite (`org.xerial:sqlite-jdbc`) do `build.gradle`
- Adiciona dependência PostgreSQL JDBC (`org.postgresql:postgresql`) ao `build.gradle`
- Adiciona Flyway ao `build.gradle` (`org.flywaydb:flyway-core`)
- Cria migrations Flyway em `src/main/resources/db/migration/`
- Atualiza `application.properties` base com datasource PostgreSQL
- Atualiza `application-dev.properties` (YAML pós-US-03.04) com URL do Docker Compose
- Atualiza `application-test.properties` para H2 in-memory com modo PostgreSQL
- Garante que toda a suíte de 73 testes passa com H2 in-memory

### ❌ O que esta história NÃO faz

- Não sobe o PostgreSQL — isso é US-03.04 (Docker Compose)
- Não migra para JPA — isso é EPIC-05
- Não configura backups ou réplicas — fora do escopo
- Não remove o arquivo `dev.db` do histórico do git — isso é limpeza futura
