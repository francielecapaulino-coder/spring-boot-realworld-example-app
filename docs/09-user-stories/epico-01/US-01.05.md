## O que é a Definition of Done

A Definition of Done define o conjunto de critérios que uma **história** precisa atender
**antes de ser fechada e a PR mergeada**. É a checagem de saída — o equivalente a confirmar
que a obra foi entregue, testada e documentada antes de assinar o recebimento.

> **Regra fundamental:** a PM não aprova PR com itens obrigatórios não marcados.  
> PR com DoD incompleta = PR não mergeada.  
> História sem PR = história não fechada.

---

## Gates obrigatórios — bloqueiam o merge independentemente de tudo

Estes 5 itens são bloqueadores absolutos para qualquer tipo de história.
PR com qualquer um destes itens não atendido **não será aprovada**:

````
🚫 GATE 1: ./gradlew test passando sem falhas (quando há código Java)
🚫 GATE 2: ./gradlew pitest ≥ 95% mantido (quando há mudança de código Java)
🚫 GATE 3: commit referencia a issue — closes #XX ou refs #XX
🚫 GATE 4: link do Coda preenchido na PR (prompts documentados)
🚫 GATE 5: PR template preenchido — todos os campos obrigatórios marcados
````

## Critérios universais — aplicam-se a TODAS as histórias

### 1. Código e funcionalidade

- [ ] O que foi implementado corresponde ao escopo da issue (nem mais, nem menos)
- [ ] Comportamento esperado verificado manualmente seguindo os critérios de aceitação da issue
- [ ] Nenhuma regressão introduzida — funcionalidades existentes continuam funcionando
- [ ] `docker compose up` continua funcional após as mudanças — todos os 6 serviços healthy:
  - app (porta 8080), postgres (5432), prometheus (9090), loki (3100), tempo (3200), grafana (3000)
- [ ] `curl http://localhost:8080/tags` continua retornando JSON válido
- [ ] `curl http://localhost:8080/actuator/health` continua retornando `{"status":"UP"}`

### 2. Processo e rastreabilidade

- [ ] Todos os commits desta branch seguem Conventional Commits — commitlint no CI valida automaticamente
- [ ] Pelo menos um commit referencia a issue com `closes #XX` (fecha ao mergear) ou `refs #XX` (referencia sem fechar)
- [ ] Issue vinculada existe, está preenchida e a DoR foi atendida antes do início do desenvolvimento
- [ ] Branch nomeada corretamente: `tipo/us-XX.XX-descricao-curta`
- [ ] PR aberta usando o template `.github/PULL_REQUEST_TEMPLATE.md`

### 3. Documentação no Coda

- [ ] Todos os prompts usados durante o desenvolvimento desta história documentados no Coda
- [ ] Skills identificadas ou aprendidas nesta história registradas no Coda
- [ ] Link do Coda preenchido no campo "Link do Coda" da PR
- [ ] Link do Coda preenchido no campo "Prompts e skills — Coda" da issue

## Critérios adicionais — por tipo de história

### Para histórias do tipo `feat` — nova funcionalidade

- [ ] `./gradlew test` passando — 100% verde
- [ ] Testes de unidade escritos para lógica nova nas camadas `core` e `application`
- [ ] Testes de integração escritos ou verificados quando há mudança de contrato de API
  - Usar REST Assured + Testcontainers PostgreSQL 16 (NUNCA H2 ou SQLite)
  - Cobrir: ≥ 1 cenário de sucesso + ≥ 1 cenário de erro por endpoint afetado
- [ ] `./gradlew pitest` — mutation score ≥ 95% mantido para todo o codebase
  - Verificar: `build/reports/pitest/index.html`
- [ ] OpenAPI/Swagger atualizado quando há novo endpoint ou campo (EPIC-11)
  - Verificar: `http://localhost:8080/swagger-ui.html` mostra o novo contrato
- [ ] Schema GraphQL `.graphqls` atualizado quando há nova query, mutation ou campo
  - Verificar: `POST /graphql` com a nova operação retorna resultado correto
- [ ] `API-mapping.md` v2.0 continua preciso após esta mudança
- [ ] Fluxo E2E no Playwright atualizado no branch `feat/playwright-working` quando há mudança de comportamento visível ao consumidor da API (EPIC-09)

### Para histórias do tipo `fix` — correção de bug

- [ ] Bug corrigido e verificado localmente seguindo os passos de reprodução da issue
- [ ] Teste de regressão adicionado — falha na branch `master` antes do fix, passa após o fix
- [ ] `./gradlew test` passando — 100% verde
- [ ] `./gradlew pitest` ≥ 95% mantido
- [ ] Nenhum comportamento correto alterado — apenas o bug corrigido
- [ ] Contrato HTTP preservado (ex: DELETE continua retornando 204 — não 200 — conforme ADR implícito e `API-mapping.md` v2.0)

### Para histórias do tipo `test` — testes

- [ ] Tipo de teste implementado conforme definido na issue: unitário | slice | integração | mutação | E2E
- [ ] **Para testes de unidade e slice:**
  - `./gradlew test` passando
  - `./gradlew pitest` ≥ 95% — novos testes aumentam ou mantêm o score
- [ ] **Para testes de integração:**
  - PostgreSQL 16 real via Testcontainers (verificar: log de teste mostra container iniciando)
  - Nenhum teste usa H2 ou SQLite: `grep -r "H2\|sqlite\|DataSource.*embedded" src/test/` → 0 resultados
  - ≥ 1 cenário de sucesso + ≥ 1 cenário de erro por endpoint ou operação coberta
- [ ] **Para testes de mutação (Pitest):**
  - `./gradlew pitest` — score ≥ 95%
  - CI configurado: threshold bloqueia merge se score < 95%
  - Relatório publicado como artefato no CI
- [ ] **Para testes E2E (Playwright):**
  - `npx playwright test` passando no branch `feat/playwright-working`
  - Branch `feat/playwright-broken` existe com CI vermelho (documentação do estado anterior)
  - Fluxos testados usando `APIRequestContext` — nunca automação de browser
  - `BASE_URL` configurável via variável de ambiente

### Para histórias do tipo `refactor` — reestruturação

- [ ] Comportamento externo idêntico ao anterior — nenhum endpoint muda resposta
- [ ] `./gradlew test` passando sem alteração nos testes existentes
- [ ] Testes de contrato (EPIC-08) passando sem qualquer modificação
- [ ] `./gradlew pitest` ≥ 95% mantido
- [ ] ADRs consultados estão listados na seção de ADRs da PR
- [ ] Nenhum arquivo do pacote `io.spring.graphql` editado manualmente (ADR-004)
  - Verificar: `git diff --name-only | grep "io/spring/graphql"` → 0 resultados

### Para histórias do tipo `docs` — documentação

- [ ] Arquivo criado ou atualizado e acessível no repositório no path correto
- [ ] Conteúdo revisado pela PM
- [ ] Links internos verificados — referências a outros documentos do projeto funcionam
- [ ] Se ADR: seção de "Rastreabilidade" com referências a iniciativas, épicos e histórias afetadas
- [ ] Se OpenAPI: `http://localhost:8080/swagger-ui.html` renderiza sem erros e JSON válido em `/v3/api-docs`

### Para histórias do tipo `spike` — investigação

- [ ] Pergunta da issue respondida com evidências concretas (não apenas opinião)
- [ ] Resultado documentado no artefato definido na issue (ADR atualizado | comentário | relatório)
- [ ] Timebox respeitado — se não concluiu no tempo, documentar o que faltou e por quê
- [ ] Próximos passos definidos com base no resultado
- [ ] Se o resultado impactar um ADR: ADR-00X atualizado em `06-architecture-decisions.md` v1.0

### Para histórias do tipo `chore` — setup e configuração

- [ ] Configuração funcionando no ambiente local e no CI
- [ ] `docker compose up` continua funcional (quando a mudança envolve Docker)
- [ ] Impacto no build Java/Gradle verificado — `./gradlew build` sem erros
- [ ] Se nova variável de ambiente adicionada: `.env.example` atualizado com a variável e comentário explicativo
- [ ] `CONTRIBUTING.md` atualizado se o setup de ambiente mudou

## Como a DoD se encaixa no fluxo de trabalho

````
Desenvolvimento concluído localmente
         ↓
Dupla verifica todos os critérios desta DoD
         ↓
./gradlew test → 100% verde?        NÃO → corrigir antes de continuar
         ↓ SIM
./gradlew pitest ≥ 95%?             NÃO → adicionar testes antes de continuar
         ↓ SIM
Coda com prompts documentados?      NÃO → documentar antes de continuar
         ↓ SIM
PR aberta com template preenchido
         ↓
PM revisa → todos os gates OK?      NÃO → dupla corrige e notifica novamente
         ↓ SIM
PR aprovada → merge na master
         ↓
bleeding branch atualizado automaticamente
         ↓
Issue fechada com closes #XX
````

## O que NÃO é requisito para fechar uma história

Para evitar over-engineering, estes itens **não são** requisitos de DoD neste projeto:

| Item | Por que NÃO é requisito |
|---|---|
| 100% de mutation score no Pitest | Custo exponencial; 95% é o ponto de equilíbrio definido nos OKRs |
| Testes de performance (JMeter, Gatling) | Fora do escopo desta modernização |
| Todos os cenários possíveis nos testes E2E | 5 fluxos críticos cobrem os cenários de maior risco |
| GraalVM Native Image | Complexidade sem benefício para este projeto |
| Testes de segurança (penetration testing) | Fora do escopo desta modernização |
| H2 como banco alternativo nos testes | H2 está proibido — apenas PostgreSQL via Testcontainers |
