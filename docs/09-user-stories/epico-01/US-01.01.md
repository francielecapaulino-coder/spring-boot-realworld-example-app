## História de usuário

**Como** membro do time de desenvolvimento,  
**quero** que o CI rejeite automaticamente qualquer commit fora do padrão Conventional Commits,  
**para que** o histórico seja sempre legível, rastreável e auditável — independentemente de quem gerou o código ou como.

---

## O que precisa existir ao final desta história

Ao concluir esta história, o repositório deve ter:

1. Um arquivo `.commitlintrc.yml` na raiz configurando as regras
2. Um `package.json` na raiz com as dependências de commitlint como `devDependencies`
3. Um `package-lock.json` gerado e commitado
4. Um workflow `.github/workflows/commitlint.yml` que executa a validação no CI
5. O CI falhando automaticamente para qualquer commit fora do padrão
6. O CI passando para commits no padrão correto

## Contexto técnico completo

### Por que commitlint neste projeto

Este projeto usa vibe coding — código gerado com assistência de IA em alta velocidade. Sem gate automático de commits, cada sessão de IA pode gerar mensagens arbitrárias ("fix", "ajuste", "wip"). O commitlint no CI é o único controle que não pode ser contornado: se o commit não passa no formato, o merge é bloqueado.

### Repositório alvo

- **URL:** `https://github.com/gothinkster/spring-boot-realworld-example-app`
- **Branch principal:** `master`
- **Linguagem:** Java 100% — o `package.json` é exclusivo para ferramentas de processo e não interfere no build Gradle
- **CI existente:** GitHub Actions com workflow `Java CI` já presente em `.github/workflows/`

### Stack desta história

- Node.js 20 LTS (apenas para commitlint — não afeta o Java)
- `@commitlint/cli` versão `^19.0.0`
- `@commitlint/config-conventional` versão `^19.0.0`
- GitHub Actions `ubuntu-latest`

## Implementação passo a passo

### Passo 1 — Criar `.commitlintrc.yml` na raiz do repositório

```yaml
# .commitlintrc.yml
extends:
  - '@commitlint/config-conventional'

rules:
  type-enum:
    - 2
    - always
    - - feat
      - fix
      - chore
      - docs
      - test
      - refactor
      - ci
      - perf
      - style

  type-case:
    - 2
    - always
    - lower-case

  type-empty:
    - 2
    - never

  scope-case:
    - 2
    - always
    - lower-case

  subject-empty:
    - 2
    - never

  subject-full-stop:
    - 2
    - never
    - '.'

  subject-case:
    - 2
    - never
    - - sentence-case
      - start-case
      - pascal-case
      - upper-case

  header-max-length:
    - 2
    - always
    - 100
```

**Regras explicadas:**
- `type-enum`: apenas os 9 tipos listados são válidos — qualquer outro rejeita
- `type-case`: o tipo deve ser lowercase — `FEAT` rejeita, `feat` aceita
- `subject-empty`: a descrição não pode estar vazia
- `subject-full-stop`: a descrição não pode terminar com ponto
- `subject-case`: a descrição não pode começar com maiúscula
- `header-max-length`: toda a linha `tipo(escopo): descrição` deve ter no máximo 100 caracteres

### Passo 5 — Criar `.github/workflows/commitlint.yml`

```yaml
name: Commitlint

on:
  push:
    branches-ignore:
      - master
      - bleeding
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  commitlint:
    name: Validate commit messages
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install commitlint
        run: npm ci

      - name: Validate all commits in branch
        run: |
          npx commitlint \
            --from ${{ github.event.pull_request.base.sha || 'HEAD~1' }} \
            --to HEAD \
            --verbose
```

**Por que `fetch-depth: 0`:** o commitlint precisa comparar todos os commits da branch com a base. Sem fetch completo, o histórico fica truncado e a validação falha ou é incompleta.

**Por que ignorar `master` e `bleeding`:** commits diretos na `master` são merges já validados. O branch `bleeding` é usado para harness development automático e tem commits gerados programaticamente.

---

### Passo 6 — Testar localmente antes de commitar

```bash
# Instalar dependências
npm install

# Testar mensagem INVÁLIDA — deve retornar erro
echo "ajuste no controller" | npx commitlint
# Saída esperada: ✖ type may not be empty [type-empty]

# Testar mensagem INVÁLIDA com tipo errado
echo "update: fixes something" | npx commitlint
# Saída esperada: ✖ type must be one of [feat, fix, chore, ...]

# Testar mensagem VÁLIDA — deve retornar sucesso
echo "feat(articles): add reading time estimation" | npx commitlint
# Saída esperada: ✔ found 0 problems, 0 warnings

# Testar mensagem VÁLIDA sem escopo
echo "chore: upgrade gradle to 9.3.1" | npx commitlint
# Saída esperada: ✔ found 0 problems, 0 warnings
```

---

### Passo 7 — Commitar tudo com a mensagem correta

```bash
git add .commitlintrc.yml package.json package-lock.json .gitignore .github/workflows/commitlint.yml
git commit -m "ci(commitlint): add conventional commits validation to CI pipeline"
git push origin <branch-name>
```

**Verificar no GitHub Actions:** após o push, acessar a aba "Actions" do repositório e confirmar que o workflow "Commitlint" aparece e executa com sucesso.

---

### Passo 8 — Validar que o CI rejeita commits inválidos

Criar uma branch de teste, fazer commit com mensagem inválida e verificar que o CI falha:

```bash
git checkout -b test/commitlint-validation
git commit --allow-empty -m "testing invalid commit"
git push origin test/commitlint-validation
# Abrir PR e verificar que o check "Commitlint" falha
```

Após verificar, deletar a branch de teste:

```bash
git push origin --delete test/commitlint-validation
git checkout -
git branch -D test/commitlint-validation
```

## Referência de tipos e escopos do projeto

### Tipos válidos

| Tipo | Quando usar | Exemplo real deste projeto |
|---|---|---|
| `feat` | Nova funcionalidade visível ao usuário ou ao sistema | `feat(articles): add reading time estimation on create` |
| `fix` | Corrige um bug | `fix(auth): move jwt secret to environment variable` |
| `chore` | Setup, dependências, ferramentas, sem impacto funcional | `chore(deps): upgrade spring boot to 4.0.6` |
| `docs` | Documentação, ADRs, README, CONTRIBUTING | `docs(adr): document dgs framework decision` |
| `test` | Adiciona ou corrige testes | `test(articles): add pitest mutation coverage for reading time` |
| `refactor` | Reestrutura código sem mudar comportamento | `refactor(infrastructure): migrate mybatis to spring data jpa` |
| `ci` | Configuração de CI/CD, workflows, scripts de build | `ci(quality): add pitest threshold gate to pipeline` |
| `perf` | Melhoria mensurável de performance | `perf(app): enable virtual threads spring boot 4` |
| `style` | Formatação de código — sem mudança de lógica | `style: apply spotless google java format` |

### Escopos recomendados para este projeto

```
auth        articles    comments    tags        profiles
users       deps        ci          docker      db
api         graphql     infra       core        app
metrics     security    records     jpa         lgtm
```

O escopo é opcional. Quando presente, deve ser lowercase e corresponder ao domínio da mudança.

## Critérios de aceitação

### CA-01 — CI rejeita mensagem sem tipo
```
Input:  "ajuste no controller"
Expect: CI falha · ✖ type may not be empty
```

### CA-02 — CI rejeita tipo não permitido
```
Input:  "update: fix something"
Expect: CI falha · ✖ type must be one of [feat, fix, chore, docs, test, refactor, ci, perf, style]
```

### CA-03 — CI rejeita tipo em maiúscula
```
Input:  "FEAT: add something"
Expect: CI falha · ✖ type must be lower-case
```

### CA-04 — CI rejeita descrição começando com maiúscula
```
Input:  "feat(articles): Add reading time"
Expect: CI falha · ✖ subject must not be sentence-case
```

### CA-05 — CI rejeita descrição terminando com ponto
```
Input:  "feat(articles): add reading time."
Expect: CI falha · ✖ subject may not end with full stop
```

### CA-06 — CI aceita commit válido com escopo
```
Input:  "feat(articles): add reading time estimation"
Expect: CI passa · ✔ found 0 problems, 0 warnings
```

### CA-07 — CI aceita commit válido sem escopo
```
Input:  "chore: upgrade gradle to 9.3.1"
Expect: CI passa · ✔ found 0 problems, 0 warnings
```

### CA-08 — Todos os 9 tipos são aceitos individualmente
Cada tipo da lista (feat, fix, chore, docs, test, refactor, ci, perf, style) deve passar quando usado corretamente.

### CA-09 — Workflow visível na aba Actions do GitHub
Após o push, o workflow "Commitlint" aparece e executa na aba Actions do repositório.

### CA-10 — `package-lock.json` presente no repositório
O arquivo existe, foi gerado por `npm install` e está commitado — garante instalação determinística no CI.
