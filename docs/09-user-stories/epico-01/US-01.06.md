## Contexto técnico completo

### O que é harness development neste projeto

Harness development é a prática de dividir cada história em etapas pequenas e verificáveis, onde cada etapa concluída gera um commit automático para o branch `bleeding`. Isso cria um histórico granular e rastreável do desenvolvimento — especialmente importante em vibe coding, onde a velocidade de geração pode obscurecer o progresso real.

### Fluxo de trabalho com o branch `bleeding`

````
master (protegido — apenas via PR)
    │
    ├── chore/us-01.06-bleeding-branch-setup  ← branch de feature da dupla
    │         │
    │         ├── commit 1: chore(ci): create bleeding branch setup
    │         ├── commit 2: chore(scripts): add bleeding-commit.sh
    │         └── commit 3: docs(process): add harness development guide
    │
bleeding (recebe push direto — rastreamento contínuo)
    │
    ├── chore(ci): step 1/3 - create bleeding branch [refs #XX]
    ├── chore(scripts): step 2/3 - add bleeding-commit.sh [refs #XX]
    └── docs(process): step 3/3 - add harness development guide [refs #XX]
````

### Por que `bleeding` e não `develop`

O branch `develop` tem semântica de "próxima versão estável". O branch `bleeding` tem semântica de "desenvolvimento em progresso, pode quebrar" — é o termo correto para um branch que recebe commits contínuos de harness development sem garantia de estabilidade em cada commit individual.

### Proteção do branch `bleeding`

O branch `bleeding` não precisa ser protegido com PR obrigatório — isso quebraria o fluxo de commits automáticos. Porém, deve ter proteção mínima:
- Não pode ser deletado por push acidental
- Não requer status checks para receber push direto

Isso é configurado via GitHub Branch Protection Rules na API ou na UI do GitHub.

````bash

# Linux: https://cli.github.com/manual/installation

# Configurar proteção: não pode ser deletado, push direto permitido

  --method PUT \
  -H "Accept: application/vnd.github+json" \
  /repos/francielecapaulino-coder/spring-boot-realworld-example-app/branches/bleeding/protection \
  -f "required_status_checks=null" \
  -f "enforce_admins=false" \
  -F "required_pull_request_reviews=null" \
  -F "restrictions=null" \
  -F "allow_deletions=false" \
  -F "allow_force_pushes=false"
````

> 1. Acessar: `Settings > Branches > Add branch protection rule`
> 2. Branch name pattern: `bleeding`
> 3. Marcar: **Do not allow deletions**
> 4. Deixar desmarcado: "Require a pull request before merging" (push direto necessário)
> 5. Salvar

### Passo 4 — Criar `scripts/bleeding-commit.sh`

````bash
mkdir -p scripts
````

Conteúdo do arquivo `scripts/bleeding-commit.sh`:

````bash
#!/bin/bash
# =============================================================================
# bleeding-commit.sh
# Automatiza commits de harness development para o branch bleeding
# Uso: ./scripts/bleeding-commit.sh "tipo(escopo): descrição" "refs #XX"
#
# Exemplo:
#   ./scripts/bleeding-commit.sh \
#     "feat(articles): step 2/4 - implement calculateReadingTime method" \
#     "refs #42"
# =============================================================================

set -e  # Parar em qualquer erro

# Cores para output
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

# Validar argumentos
if [ $# -lt 2 ]; then
  echo -e "${RED}Erro: argumentos insuficientes${NC}"
  echo ""
  echo "Uso: ./scripts/bleeding-commit.sh \"tipo(escopo): descrição\" \"refs #XX\""
  echo ""
  echo "Exemplos:"
  echo "  ./scripts/bleeding-commit.sh \"feat(articles): step 1/3 - add reading time migration\" \"refs #42\""
  echo "  ./scripts/bleeding-commit.sh \"refactor(infra): step 2/5 - migrate UserMapper to JPA\" \"refs #55\""
  exit 1
fi

COMMIT_MESSAGE="$1"
ISSUE_REF="$2"
CURRENT_BRANCH=$(git branch --show-current)
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

# Validar formato da referência de issue
if [[ ! "$ISSUE_REF" =~ ^(refs|closes)\ #[0-9]+$ ]]; then
  echo -e "${RED}Erro: referência de issue inválida${NC}"
  echo "Formato esperado: 'refs #XX' ou 'closes #XX'"
  echo "Recebido: '$ISSUE_REF'"
  exit 1
fi

# Validar formato do Conventional Commit (básico)
if [[ ! "$COMMIT_MESSAGE" =~ ^(feat|fix|chore|docs|test|refactor|ci|perf|style)(\([a-z-]+\))?:\ .+ ]]; then
  echo -e "${RED}Erro: mensagem não segue Conventional Commits${NC}"
  echo "Formato esperado: 'tipo(escopo): descrição'"
  echo "Tipos válidos: feat, fix, chore, docs, test, refactor, ci, perf, style"
  echo "Recebido: '$COMMIT_MESSAGE'"
  exit 1
fi

echo -e "${YELLOW}=== Bleeding Branch Commit ===${NC}"
echo "Branch atual:  $CURRENT_BRANCH"
echo "Mensagem:      $COMMIT_MESSAGE"
echo "Issue:         $ISSUE_REF"
echo "Timestamp:     $TIMESTAMP"
echo ""

# Salvar estado atual (stash se houver mudanças não staged)
STASH_NEEDED=false
if ! git diff --quiet || ! git diff --staged --quiet; then
  echo -e "${YELLOW}Salvando estado atual com git stash...${NC}"
  git stash push -m "bleeding-commit-temp-stash-$TIMESTAMP"
  STASH_NEEDED=true
fi

# Mudar para bleeding, fazer merge do estado atual, commit e push
echo -e "${YELLOW}Mudando para branch bleeding...${NC}"
git checkout bleeding
git pull origin bleeding --rebase

echo -e "${YELLOW}Fazendo merge da branch $CURRENT_BRANCH...${NC}"
git merge "$CURRENT_BRANCH" --no-ff --no-commit 2>/dev/null || true

# Staging de todas as mudanças
git add -A

# Verificar se há algo para commitar
if git diff --staged --quiet; then
  echo -e "${YELLOW}Nada a commitar em bleeding — estado já sincronizado${NC}"
  git checkout "$CURRENT_BRANCH"
  if [ "$STASH_NEEDED" = true ]; then
    git stash pop
  fi
  exit 0
fi

# Fazer o commit no bleeding
git commit -m "$COMMIT_MESSAGE

$ISSUE_REF
bleeding-branch-auto-commit: $TIMESTAMP
source-branch: $CURRENT_BRANCH"

# Push para remoto
echo -e "${YELLOW}Push para origin/bleeding...${NC}"
git push origin bleeding

# Voltar para a branch de trabalho
git checkout "$CURRENT_BRANCH"

# Restaurar stash se necessário
if [ "$STASH_NEEDED" = true ]; then
  echo -e "${YELLOW}Restaurando estado com git stash pop...${NC}"
  git stash pop
fi

echo ""
echo -e "${GREEN}✓ Commit no bleeding concluído com sucesso!${NC}"
echo -e "${GREEN}✓ Branch de trabalho: $CURRENT_BRANCH (restaurada)${NC}"
echo ""
echo "Verifique no GitHub:"
echo "https://github.com/francielecapaulino-coder/spring-boot-realworld-example-app/commits/bleeding"
````

````bash
# Tornar o script executável
chmod +x scripts/bleeding-commit.sh
````

````markdown
Harness development é a prática de dividir cada história em **etapas pequenas e

## Como usar no dia a dia

### Formato de commit para o bleeding

Todo commit no branch `bleeding` segue Conventional Commits com indicação de etapa:

````
tipo(escopo): step X/Y - descrição da etapa

refs #XX
bleeding-branch-auto-commit: TIMESTAMP
source-branch: NOME-DA-BRANCH
````

### Usando o script automatizado

```bash
# Sintaxe
./scripts/bleeding-commit.sh "tipo(escopo): step X/Y - descrição" "refs #XX"

# Exemplos reais por tipo de história

# feat — nova funcionalidade
./scripts/bleeding-commit.sh \
  "feat(articles): step 1/4 - add reading_time_minutes flyway migration" \
  "refs #42"

./scripts/bleeding-commit.sh \
  "feat(articles): step 2/4 - implement calculateReadingTime method" \
  "refs #42"

./scripts/bleeding-commit.sh \
  "feat(articles): step 3/4 - add readingTimeMinutes to article response" \
  "refs #42"

./scripts/bleeding-commit.sh \
  "feat(articles): step 4/4 - add lazy update on GET /articles/:slug" \
  "refs #42"

# refactor — migração JPA
./scripts/bleeding-commit.sh \
  "refactor(infra): step 1/3 - remove mybatis starter from build.gradle" \
  "refs #55"

./scripts/bleeding-commit.sh \
  "refactor(infra): step 2/3 - add JPA annotations to User entity" \
  "refs #55"

./scripts/bleeding-commit.sh \
  "refactor(infra): step 3/3 - create UserRepository and remove UserMapper" \
  "refs #55"

# chore — configuração
./scripts/bleeding-commit.sh \
  "chore(docker): step 1/2 - create multi-stage Dockerfile" \
  "refs #30"

./scripts/bleeding-commit.sh \
  "chore(docker): step 2/2 - create docker-compose.yml with LGTM stack" \
  "refs #30"
```

### Quando commitar no bleeding

Commitar no `bleeding` a cada etapa concluída da história — não apenas no final.
Uma etapa é uma unidade de trabalho verificável:

| Tipo de história | Quando commitar no bleeding |
|---|---|
| `feat` | Após cada sub-tarefa: migration criada, cálculo implementado, response atualizado, lazy update |
| `fix` | Após reproduzir o bug + após corrigir + após adicionar teste de regressão |
| `test` | Após configuração inicial + após cada grupo de testes (auth, artigos, perfis) |
| `refactor` | Após cada entidade/repositório migrado |
| `docs` | Após cada seção principal documentada |
| `chore` | Após cada componente configurado |
````

````
## Resolução de problemas comuns

### Erro: "nothing to commit" no bleeding

O bleeding já está sincronizado com a branch de trabalho.
Certifique-se de ter feito mudanças na branch de trabalho antes de chamar o script.

### Erro: "merge conflict" no bleeding

```bash
# Resolver conflito manualmente
git checkout bleeding
git pull origin bleeding
git merge <sua-branch-de-trabalho>
# Resolver conflitos
git add .
git commit -m "chore(bleeding): resolve merge conflict [refs #XX]"
git push origin bleeding
git checkout <sua-branch-de-trabalho>
```

### Erro: "non-fast-forward" no push do bleeding

```bash
git checkout bleeding
git pull origin bleeding --rebase
git push origin bleeding
git checkout <sua-branch-de-trabalho>
```
````

````
ARQUIVO 1: scripts/bleeding-commit.sh
Script Bash que automatiza commits de harness development para o branch bleeding.

Requisitos do script:
- Parâmetros: $1 = mensagem do commit, $2 = referência de issue
- Validar que foram passados 2 argumentos — se não, mostrar usage e exit 1
- Validar que $2 segue formato: "refs #N" ou "closes #N" — regex: ^(refs|closes)\ #[0-9]+$
- Validar que $1 segue Conventional Commits básico — regex: ^(feat|fix|chore|docs|test|refactor|ci|perf|style)(\([a-z-]+\))?:\ .+
- Se validações falharem: mensagem de erro colorida (vermelho) + exit 1
- Fluxo principal:
  1. Salvar branch atual em variável CURRENT_BRANCH
  2. Se houver mudanças não committed (git diff não vazio): git stash push com nome temporário
  3. git checkout bleeding
  4. git pull origin bleeding --rebase
  5. git merge $CURRENT_BRANCH --no-ff --no-commit (silenciar erros de merge vazio)
  6. git add -A
  7. Verificar se há algo staged — se não houver, restaurar e exit 0 com mensagem
  8. git commit com mensagem composta:
     "$1\n\n$2\nbleeding-branch-auto-commit: TIMESTAMP\nsource-branch: $CURRENT_BRANCH"
  9. git push origin bleeding
  10. git checkout $CURRENT_BRANCH
  11. Se houve stash: git stash pop
  12. Mensagem de sucesso verde: "✓ Commit no bleeding concluído com sucesso!"
  13. Mostrar URL: https://github.com/francielecapaulino-coder/spring-boot-realworld-example-app/commits/bleeding
- Usar cores: RED='\033[0;31m' GREEN='\033[0;32m' YELLOW='\033[1;33m' NC='\033[0m'
- set -e no topo (parar em qualquer erro)

== CRIAÇÃO DO BRANCH BLEEDING ==
O branch bleeding deve ser criado ANTES de commitar os arquivos desta história.
Sequência:
1. git checkout -b bleeding
2. git push origin bleeding
3. Configurar proteção via GitHub CLI (gh api) ou documentar como fazer via UI
4. git checkout chore/us-01.06-bleeding-branch-setup

Proteção mínima necessária (via gh api ou UI):
- allow_deletions: false
- allow_force_pushes: false
- NÃO exigir PR para merge (push direto necessário para o script funcionar)
````
