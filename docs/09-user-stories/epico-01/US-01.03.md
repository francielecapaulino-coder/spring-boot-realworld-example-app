## História de usuário

**Como** Product Manager e reviewera de Pull Requests,  
**quero** que toda PR aberta no repositório exiba automaticamente um template com checklist obrigatório — vinculando a PR à issue que a originou, aos testes executados, ao Pitest, ao Coda e aos ADRs consultados —  
**para que** nenhuma PR seja mergeada sem rastreabilidade completa, sem testes verificados e sem documentação de prompts no Coda.

---

## O que precisa existir ao final desta história

1. Um arquivo `.github/PULL_REQUEST_TEMPLATE.md` na raiz do repositório
2. O template aparece automaticamente ao abrir qualquer PR no GitHub
3. O template contém:
   - Campo obrigatório de link para issue
   - Tipo de mudança (alinhado com Conventional Commits)
   - Descrição do que foi feito
   - ADRs consultados
   - Checklist de DoD espelhando `docs/process/definition-of-done.md`
   - Link obrigatório para o Coda
   - Seção de evidência de testes
4. A PM pode usar o checklist como gate de aprovação — itens não marcados = PR não aprovada

## Contexto técnico completo

### Por que um template de PR e não apenas o template de issue

A issue documenta **o que** vai ser feito e **por que**. A PR documenta **o que foi feito**, **como foi feito**, **quais testes cobrem** e **onde está a documentação no Coda**. São artefatos complementares — a issue é o briefing de entrada, a PR é o relatório de entrega.

Sem o template de PR, o reviewer precisa adivinhar qual issue motivou o código, se os testes foram executados, se o Pitest continua no threshold e se os prompts foram documentados. O template elimina essa adivinhação.

### Diferença de GitHub Issue Forms (YAML) e PR Template (Markdown)

GitHub Issue Forms (`.yml`) suportam campos obrigatórios com validação no lado do servidor — por isso os templates de issue são YAML.

GitHub Pull Request Templates (`.md`) são **Markdown puro** — não suportam campos obrigatórios com validação automática. A "obrigatoriedade" é enforçada pelo **processo de review**: a PM não aprova PR com checklist incompleto.

### Estrutura de arquivos

│   ├── config.yml

## Critérios de aceitação

### CA-01 — Template aparece automaticamente ao abrir qualquer PR
**Dado que** o arquivo `.github/PULL_REQUEST_TEMPLATE.md` está commitado no branch `master`,  
**quando** qualquer pessoa abre uma nova PR no repositório,  
**então** o corpo da PR vem pré-preenchido com o template completo.

### CA-02 — Campo "Issue relacionada" presente e visível
O template contém `closes #` no topo como primeiro campo editável. A PM verifica que o campo foi preenchido durante o review.

### CA-03 — Checklist de tipo de mudança com todos os 9 tipos
Os 9 tipos do Conventional Commits aparecem como checkboxes: `feat`, `fix`, `test`, `docs`, `chore`, `refactor`, `ci`, `perf`, `style`.

### CA-04 — Checklist de DoD completo com itens condicionais
O checklist inclui:
- Itens de código e funcionalidade (3 itens)
- Itens de testes com `./gradlew test`, `./gradlew pitest`, Playwright e Testcontainers (6 itens)
- Itens de contrato: OpenAPI, schema GraphQL, API Mapping (3 itens)
- Itens de processo: Conventional Commits, issue link, Coda (4 itens)

### CA-05 — Referência à fonte de verdade do DoD
O template contém a linha: `<!-- Fonte de verdade: docs/process/definition-of-done.md -->`.

### CA-06 — Seção de ADRs com os 6 ADRs do projeto
Os 6 ADRs de `06-architecture-decisions.md` v1.0 aparecem como checkboxes:
- ADR-001 (DGS Framework), ADR-002 (JPA queries), ADR-003 (AOP métricas), ADR-004 (código gerado), ADR-005 (Node/records), ADR-006 (JWT env vars)

### CA-07 — Campo "Link do Coda" presente
O campo existe no template com instrução de preenchimento obrigatório.

### CA-08 — Seções collapsible para evidência de testes
Três seções `<details>` existem: `./gradlew test`, `./gradlew pitest`, Playwright.

### CA-09 — Template é Markdown puro (não YAML)
O arquivo é `.md` e não `.yml` — GitHub PR templates são obrigatoriamente Markdown.

### CA-10 — Primeiro uso real: PR desta história preenche o template
A PR que entrega esta história é o primeiro uso do template como exemplo real.
