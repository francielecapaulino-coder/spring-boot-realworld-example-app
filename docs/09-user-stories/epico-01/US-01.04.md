## Contexto técnico completo

### O que é Definition of Ready neste projeto

Definition of Ready é o conjunto de critérios que uma **issue** precisa atender antes de poder entrar em desenvolvimento. É a checagem de entrada — o equivalente a confirmar que todos os ingredientes estão na bancada antes de começar a cozinhar.

Neste projeto, a DoR é especialmente crítica porque:

1. **Vibe coding com IA:** se o contexto estiver incompleto, a LLM gera código inconsistente com os ADRs e documentos do projeto
2. **Múltiplas duplas em paralelo:** o time precisa de critérios objetivos e compartilhados para que diferentes duplas apliquem o mesmo padrão
3. **Harness development:** commits automáticos para o branch `bleeding` exigem que a branch esteja corretamente configurada desde o início
4. **Ambiente Docker:** sem ambiente funcionando, o desenvolvimento não pode ser verificado localmente

## O que é a Definition of Ready

A Definition of Ready define o conjunto de critérios que uma **issue** precisa atender
**antes de entrar em desenvolvimento**. Nenhuma dupla inicia o trabalho em uma história
sem que todos os critérios aplicáveis estejam verificados.

> **Regra fundamental:** código só é gerado após a issue estar pronta.  
> Issue não pronta = desenvolvimento não começa.  
> Sem issue = desenvolvimento não começa.

---

## Critérios adicionais — por tipo de história

### Para histórias do tipo `feat` ou `fix` (código de produção)

- [ ] `./gradlew test` passando localmente na master antes de iniciar
- [ ] `./gradlew pitest` configurado e com threshold de 95% ativo no CI
- [ ] Testcontainers PostgreSQL disponível (Docker rodando)
- [ ] Nenhuma referência a H2 ou SQLite nos testes existentes (`grep -r "H2\|sqlite" src/test/ → 0`)
- [ ] Entidade ou serviço afetado mapeado — camadas a tocar identificadas:
  `api` | `core` | `application` | `infrastructure`

### Para histórias do tipo `test` (integração, mutação, E2E)

- [ ] Tipo de teste definido na issue: unitário | slice | integração | mutação | E2E
- [ ] Para integração: PostgreSQL 16 disponível via Testcontainers
- [ ] Para mutação (Pitest): `./gradlew pitest` executável localmente
- [ ] Para E2E (Playwright): Node.js instalado, `npx playwright test` executável
- [ ] Para Playwright: branch `feat/playwright-broken` existe (documentação do problema)
- [ ] Endpoints ou operações a testar listados na issue (ex: "19/19 REST" ou "fluxo 1, 2, 3")

### Para histórias do tipo `refactor`

- [ ] ADRs que definem como a refatoração deve ser feita estão na issue (campo obrigatório)
- [ ] Testes de contrato (EPIC-08) passando antes de iniciar — verificar que existem
- [ ] Comportamento atual documentado na issue (campo "Estado atual do código")
- [ ] Comportamento alvo documentado na issue (campo "Estado alvo do código")

### Para histórias do tipo `docs`

- [ ] Artefato a ser criado ou atualizado identificado (path exato do arquivo)
- [ ] Referências cruzadas com outros documentos verificadas
  (ex: US-01.04 referencia `05-backlog.md`, `06-architecture-decisions.md`, etc.)

### Para histórias do tipo `spike`

- [ ] Pergunta a ser respondida é mensurável (não apenas "investigar X")
- [ ] Timebox definido na issue — ao expirar, parar e documentar o que foi descoberto
- [ ] Entregável definido: ADR atualizado | comentário na issue | relatório em `docs/`

### Para histórias do tipo `chore` (inclui `ci`, `perf`, `style`)

- [ ] Escopo claro — o que muda e o que NÃO muda está descrito
- [ ] Impacto no build Java/Gradle verificado (ex: `package.json` não afeta Gradle)
- [ ] Variáveis de ambiente necessárias documentadas em `.env.example`

## Como a DoR se encaixa no fluxo de trabalho

````
1. PM cria ou aprova a issue no GitHub usando template de .github/ISSUE_TEMPLATE/
         ↓
2. Dupla verifica DoR — todos os critérios desta página atendidos
         ↓
3. Coda aberto — prompt preparado com contexto dos ADRs e épico
         ↓
4. Branch criada: tipo/us-XX.XX-descricao
         ↓
5. Vibe coding inicia — código gerado com contexto completo
         ↓
6. DoD verificada (docs/process/definition-of-done.md)
         ↓
7. PR aberta usando .github/PULL_REQUEST_TEMPLATE.md
         ↓
8. PM revisa e aprova
         ↓
9. Merge → bleeding branch atualizado automaticamente
````
