## História de usuário

**Como** mantenedor da camada de dados da aplicação,  
**quero** converter os wrappers e contadores do pacote `application.data` para tipos `record`,  
**para que** o código seja mais conciso, imutável por padrão e o Lombok seja removido dessas classes.

---

## Contexto

US-06.01 converteu os DTOs de leitura principais (`ProfileData`, `ArticleData`, `CommentData`). Restam no pacote `io.spring.application.data` quatro classes Lombok que são candidatas diretas:

| Classe atual | Anotações Lombok | Por que é candidata a record |
|---|---|---|
| `UserData` | `@Data @NoArgsConstructor @AllArgsConstructor` | 5 campos `String`, sem comportamento, sem herança |
| `UserWithToken` | `@Getter`, construtor manual que expande `UserData` | Wrapper de resposta; imutável por design |
| `ArticleDataList` | `@Getter`, construtor manual | Wrapper de lista + count; imutável por design |
| `ArticleFavoriteCount` | `@Value` (Lombok — similar a record) | 2 campos (`id`, `count`); já é imutável |

Nenhuma dessas classes implementa interface com método `getCursor()` (diferente de `ArticleData` e `CommentData`), o que torna a conversão mais simples.

**Cuidado com `UserWithToken`:** o construtor atual expande `UserData` (recebe `UserData` + token e copia campos). No record, o construtor canônico recebe os 5 campos diretamente — um construtor compacto secundário pode ser adicionado para compatibilidade com o código que ainda cria `UserWithToken(userData, token)`.

---

## Escopo

* Converter `UserData` para record: `public record UserData(String id, String email, String username, String bio, String image) {}`
* Converter `ArticleFavoriteCount` para record: `public record ArticleFavoriteCount(String id, Integer count) {}`
* Converter `ArticleDataList` para record — manter `@JsonProperty("articles")` e `@JsonProperty("articlesCount")` nos componentes, ou usar getters customizados
* Converter `UserWithToken` para record — adicionar construtor compacto secundário `UserWithToken(UserData, String)` que delega ao canônico
* Remover anotações Lombok (`@Data`, `@Getter`, `@Value`, `@NoArgsConstructor`, `@AllArgsConstructor`) das classes convertidas
* Garantir que todos os controllers e serviços que instanciam essas classes continuem compilando

## Fora de escopo

* `ArticleData` e `CommentData` — cobertos por US-06.01
* `ProfileData` — coberto por US-06.01
* Request params / write DTOs — cobertos por US-06.03
* Remoção do Lombok de classes **não** elegíveis (`Page`, `CursorPageParameter`) — fora do critério de elegibilidade

## Critérios de aceite

* [ ] `UserData`, `ArticleFavoriteCount`, `ArticleDataList`, `UserWithToken` são `record` e compilam
* [ ] Nenhuma dessas classes tem import `lombok.*`
* [ ] `./gradlew build` sem warnings
* [ ] `./gradlew test` 100% verde
* [ ] Serialização Jackson continua produzindo o mesmo JSON (verificar com testes de integração existentes)

## Regras técnicas

* Preservar `@JsonProperty` nos componentes do record onde necessário (especialmente `ArticleDataList`)
* Em `UserWithToken`: o construtor canônico deve ter os 5 campos (`email`, `username`, `bio`, `image`, `token`); o construtor de conveniência `UserWithToken(UserData, String)` pode ser definido dentro do corpo do record como construtor customizado delegando ao `this(...)`
* `@Value` do Lombok em `ArticleFavoriteCount` é equivalente a record — substituição direta sem edge cases

## Dependências

Os quatro wrappers/contadores do pacote `data` são records imutáveis sem Lombok. Código mais conciso e comportamento externo (JSON da API) idêntico.
