## História de usuário

**Como** mantenedor da camada de domínio,  
**quero** converter value objects do `core` que não são entidades JPA para tipos `record`,  
**para que** o domínio tenha objetos de valor imutáveis por padrão, sem boilerplate.

---

## Contexto

No pacote `io.spring.core` existem classes que representam value objects do domínio — não entidades gerenciadas pelo JPA. A elegibilidade depende do resultado de EPIC-05:

| Classe | Anotações atuais | Elegibilidade (pré-EPIC-05) | O que verificar após EPIC-05 |
|---|---|---|---|
| `Tag` | `@Data @NoArgsConstructor @EqualsAndHashCode(of = "name")` com construtor que gera UUID | ⚠️ Condicional | Se EPIC-05 adicionou `@Entity` → **não converter**. Se permanece como value object (tag como `String` ou mapeada de outra forma) → converter |
| `FollowRelation` | `@Data @NoArgsConstructor`, 2 campos `String` (`userId`, `targetId`) | ⚠️ Condicional | Se EPIC-05 adicionou `@Entity` → **não converter**. Se permanece como value object sem `@Entity` → converter |

**Cenário mais provável após EPIC-05:**
- `Tag` mapeada como entidade `@Entity` para a tabela `tags` → **inelegível**
- `FollowRelation` mapeada como entidade `@Entity` para a tabela `follow_relations` → **inelegível**

**Se ambas forem entidades após EPIC-05**, esta história documenta a análise e fecha sem conversão de código — com justificativa explícita. O critério de elegibilidade do INI-06 proíbe conversão de classes `@Entity`.

**Cenário alternativo:** se `Tag` em EPIC-05 for representada como `String` embedded em `Article` (via `@ElementCollection` ou similar), ela pode não ter `@Entity` própria — nesse caso, o value object `Tag.java` atual seria removido ou reduzido, e não há nada a converter.

---

## Escopo

* **Passo 1 (análise):** após EPIC-05 concluído, verificar se `Tag` e `FollowRelation` têm `@Entity`. Documentar resultado nesta história.
* **Passo 2a (se não @Entity):** converter as classes elegíveis para record, removendo Lombok
  * `Tag`: `public record Tag(String id, String name) {}` com factory method estático `of(String name)` gerando UUID para substituir o construtor com UUID
  * `FollowRelation`: `public record FollowRelation(String userId, String targetId) {}`
* **Passo 2b (se @Entity):** documentar justificativa, fechar história sem conversão de código, atualizar `05-backlog.md`

## Fora de escopo

* Entidades JPA com `@Entity` — critério de elegibilidade do INI-06 as exclui explicitamente
* `Article`, `Comment`, `User`, `ArticleFavorite` — entidades do domínio com `@Entity`; **nunca converter para record**
* Mudança na lógica de geração de UUID (se `Tag` for convertida, o UUID continua sendo gerado — apenas via factory method em vez de construtor)

## Critérios de aceite

**Se conversão ocorrer:**
* [ ] Classes convertidas são `record` e compilam sem `@Entity`
* [ ] `./gradlew test` 100% verde
* [ ] Nenhuma das classes convertidas tem import `lombok.*`
* [ ] `EqualsAndHashCode(of = "name")` de `Tag`: em records, `equals/hashCode` usa todos os componentes por padrão — se o comportamento anterior era baseado apenas em `name`, implementar `equals/hashCode` customizados no corpo do record

**Se conversão não ocorrer (classes são @Entity após EPIC-05):**
* [ ] Justificativa documentada neste arquivo e no código com comentário
* [ ] `05-backlog.md` atualizado com status e motivo

## Regras técnicas

* Verificar se `Tag.java` ainda existe como classe própria após EPIC-05 (pode ter sido removida se tags viram `String` embedded)
* Se `Tag` for convertida: `@EqualsAndHashCode(of = "name")` do Lombok significa que dois `Tag` com o mesmo `name` são iguais — em records, todos os componentes participam do `equals`. Implementar `equals/hashCode` explicitamente se for necessário manter o comportamento anterior
* Factory method sugerido para `Tag` (substitui construtor com UUID): `public static Tag of(String name) { return new Tag(UUID.randomUUID().toString(), name); }`
