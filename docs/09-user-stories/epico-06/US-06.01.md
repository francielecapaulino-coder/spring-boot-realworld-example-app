## História de usuário

**Como** mantenedor do modelo de dados da aplicação,
**quero** converter DTOs de leitura sem comportamento para tipos `record`,
**para que** o código seja mais conciso, imutável por padrão e alinhado ao Java 25.

---

## Contexto

O pacote `io.spring.application.data` contém classes Lombok que são apenas portadores de dados imutáveis. Com Java 25, são candidatas diretas a `record`.

| Classe | Anotações Lombok | Observação crítica |
|---|---|---|
| `ProfileData` | `@Getter @AllArgsConstructor` | Sem comportamento; conversão direta |
| `ArticleData` | `@Getter @AllArgsConstructor` | ⚠️ **ADR-005**: implementa `Node` com `getCursor()` — método deve ser declarado explicitamente no corpo do record |
| `CommentData` | `@Getter @AllArgsConstructor` | Contém `ProfileData` como componente; converter após `ProfileData` |

**ADR-005 (obrigatório):** `ArticleData` implementa a interface `Node`. Records não herdam comportamento de interfaces — `getCursor()` deve ser declarado explicitamente: `@Override public String getCursor() { return id(); }`. Ignorar isso quebra o GraphQL.

---

## Escopo

* Converter `ProfileData` para record: `public record ProfileData(String username, String bio, String image, boolean following) {}`
* Converter `ArticleData` para record — declarar `getCursor()` explicitamente no corpo do record (ADR-005)
* Converter `CommentData` para record após `ProfileData`
* Remover anotações Lombok das três classes
* Verificar e manter `@JsonProperty`, `@JsonInclude`, `@JsonIgnoreProperties` onde existirem nos componentes
* Garantir que controllers REST e resolvers GraphQL/DGS continuam compilando

## Fora de escopo

* Entidades JPA (`Article`, `Comment`, `User`) — nunca elegíveis
* Wrappers e contadores (`UserData`, `ArticleDataList`, etc.) — cobertos por US-06.02
* Write DTOs e commands — cobertos por US-06.03

## Critérios de aceite

* [ ] `ProfileData`, `ArticleData`, `CommentData` são `record` e compilam
* [ ] `ArticleData` declara `getCursor()` explicitamente (ADR-005)
* [ ] Nenhuma das três classes tem `import lombok.*`
* [ ] `./gradlew build` sem warnings
* [ ] `./gradlew test` 100% verde
* [ ] Resolvers GraphQL/DGS continuam funcionando (campos de `ArticleData` acessíveis)

## Regras técnicas

* `getCursor()` em `ArticleData` é obrigatório: `@Override public String getCursor() { return id(); }` dentro do corpo do record
* Anotações Jackson (`@JsonProperty`, `@JsonInclude`) posicionadas nos componentes do record
* Converter na ordem: `ProfileData` → `CommentData` → `ArticleData` (menor para maior dependência)

## Dependências

* EPIC-04 concluído (Java 25 + Spring Boot 4.0.3)

## Resultado esperado

Os três DTOs de leitura principais são records imutáveis sem Lombok. Padrão de conversão estabelecido para US-06.02 seguir.
