## História de usuário

**Como** mantenedor da camada de API da aplicação,  
**quero** converter os write DTOs e commands elegíveis para tipos `record`,  
**para que** o código de entrada da API também seja conciso, imutável por padrão e sem boilerplate Lombok.

---

## Contexto

Os DTOs de escrita (request params) usam Lombok `@Getter` + `@NoArgsConstructor` + `@AllArgsConstructor` para deserialização Jackson com `@JsonRootName`. Com Java 25 e Spring Boot 4, Jackson 3.x suporta deserialização de records nativamente — sem precisar de `@NoArgsConstructor`. O desafio técnico é garantir que:
1. `@JsonRootName` continue funcionando
2. Anotações Bean Validation (`@NotBlank`, `@Email`, `@DuplicatedEmailConstraint`) funcionem nos componentes do record
3. A deserialização de JSON aninhado funcione corretamente

**Mapeamento de candidatos:**

| Classe atual | Anotações Lombok | Candidata? | Cuidados |
|---|---|---|---|
| `NewArticleParam` | `@Getter @NoArgsConstructor @AllArgsConstructor @Builder` | ✅ Sim | `@Builder` perde valor como record (já tem construtor canônico); remover `@Builder`. `@DuplicatedArticleConstraint` no campo `title` deve continuar funcionando |
| `RegisterParam` | `@Getter @NoArgsConstructor @AllArgsConstructor` | ✅ Sim | `@NotBlank`, `@Email`, `@DuplicatedEmailConstraint` nos campos — verificar que Jackson 3 + Bean Validation aplica restrições nos componentes do record |
| `UpdateUserCommand` | `@Getter @AllArgsConstructor` | ✅ Sim | Apenas 2 campos (`User targetUser`, `UpdateUserParam param`); simples holder; manter `@UpdateUserConstraint` na class-level |
| `UpdateArticleParam` | `@Getter @NoArgsConstructor @AllArgsConstructor` | ⚠️ Parcial | Campos têm default `= ""` — records não suportam valores default nos componentes. Usar compact constructor que aplica defaults, ou manter como classe |
| `UpdateUserParam` | `@Builder @Getter` com `@Builder.Default` | ❌ Não | `@Builder.Default` implica estado mutável por construção; manter como classe Lombok |

## Escopo

* Converter `NewArticleParam` para record — remover `@Builder`, manter `@DuplicatedArticleConstraint` no componente `title`
* Converter `RegisterParam` para record — verificar que Bean Validation funciona nos componentes do record com Spring Boot 4
* Converter `UpdateUserCommand` para record — substituição direta, manter `@UpdateUserConstraint` na declaração do record
* Avaliar `UpdateArticleParam`: se compact constructor com defaults funcionar corretamente com Jackson 3 + Spring MVC 4, converter; caso contrário, documentar a exclusão com justificativa e manter como classe
* Remover Lombok das classes convertidas
* Garantir que controllers e serviços que instanciam esses objetos continuem compilando

## Fora de escopo

* `UpdateUserParam` — `@Builder.Default` implica mutabilidade; manter como classe Lombok (exclusão justificada nos critérios de elegibilidade do INI-06)
* Alterações nas constraints de validação (`@DuplicatedEmailConstraint`, `@DuplicatedArticleConstraint`) — apenas reposicionamento nos componentes do record, sem mudança de lógica
* Mudança na lógica de deserialização Jackson — apenas adaptação estrutural da classe

## Critérios de aceite

* [ ] `NewArticleParam`, `RegisterParam`, `UpdateUserCommand` são `record` e compilam
* [ ] `POST /users` com body inválido continua retornando 422 com erros de validação (Bean Validation funciona no record)
* [ ] `POST /articles` com título duplicado continua retornando 422 (`@DuplicatedArticleConstraint` funciona no record)
* [ ] Nenhuma das classes convertidas tem import `lombok.*`
* [ ] `./gradlew build` sem warnings
* [ ] `./gradlew test` 100% verde
* [ ] `UpdateArticleParam`: se mantida como classe, o arquivo `US-06.03.md` documenta a justificativa; se convertida, testes confirmam defaults corretos

## Regras técnicas

* Em records com `@JsonRootName`, verificar que `JacksonCustomizations` e `DeserializationFeature.UNWRAP_ROOT_VALUE` ainda funcionam — testar com chamada real de API
* Bean Validation em records: as anotações devem ser posicionadas diretamente nos componentes do record (antes do tipo ou no mesmo local do parâmetro do construtor canônico)
* `@UpdateUserConstraint` é class-level annotation — pode ser mantida antes da declaração `public record UpdateUserCommand(...)`
* Se `UpdateArticleParam` for mantida como classe: documentar explicitamente no código com comentário `// record inelegível: campos com defaults implícitos — ver US-06.03`

Write DTOs elegíveis convertidos para records, mantendo comportamento de validação e deserialização idêntico. `UpdateUserParam` permanece como classe Lombok com justificativa documentada.
