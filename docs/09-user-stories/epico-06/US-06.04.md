## História de usuário

**Como** mantenedor da camada de tratamento de erros da API,  
**quero** converter os recursos de exceção para tipos `record`,  
**para que** as estruturas de erro sejam imutáveis por padrão e sem boilerplate Lombok.

---

## Contexto

No pacote `io.spring.api.exception` há duas classes que representam a estrutura de erros retornada pela API:

| Classe atual | Estrutura | Candidata? | Observações |
|---|---|---|---|
| `FieldErrorResource` | `@Getter @AllArgsConstructor` com 4 `String` (`resource`, `field`, `code`, `message`) | ✅ Sim | Estrutura pura de dados; sem comportamento; imutável por design |
| `ErrorResource` | `@Getter` com `List<FieldErrorResource>` e custom constructor; tem `@JsonSerialize(ErrorResourceSerializer.class)` | ⚠️ Avaliar | Tem serializer customizado — verificar se `@JsonSerialize` funciona em record da mesma forma |

**Sobre `ErrorResource`:** a classe usa `@JsonIgnoreProperties` e `@JsonRootName("errors")` além do serializer customizado `ErrorResourceSerializer`. Se o serializer referencia `fieldErrors` por getter (`getFieldErrors()`), no record a referência muda para acesso direto ao componente (`fieldErrors()`). Verificar se `ErrorResourceSerializer` usa `getter` ou acesso direto — ajustar se necessário.

---

## Escopo

* Converter `FieldErrorResource` para record: `public record FieldErrorResource(String resource, String field, String code, String message) {}`
* Avaliar `ErrorResource`: verificar se `ErrorResourceSerializer` acessa o campo via getter ou reflexão
  * Se via getter (`getFieldErrors()`): o record expõe `fieldErrors()` — ajustar o serializer para usar `fieldErrors()` em vez de `getFieldErrors()`
  * Se via reflexão de campo ou Jackson property: a conversão é direta
* Converter `ErrorResource` para record após verificação, mantendo todas as anotações Jackson
* Remover Lombok de ambas as classes convertidas
* Garantir que respostas de erro (`422`) continuam com formato JSON correto

## Fora de escopo

* Mudanças na lógica de tratamento de exceções (`CustomizeExceptionHandler`)
* Mudança no formato de resposta de erro da API
* Outras classes do pacote exception (`InvalidAuthenticationException`, `InvalidRequestException`, etc.) — são exceptions, não DTOs

## Critérios de aceite

* [ ] `FieldErrorResource` é `record` e compila
* [ ] `ErrorResource` é `record` (ou mantida como classe com justificativa documentada) e compila
* [ ] `POST /users` com body inválido continua retornando JSON de erro no formato `{"errors": {"body": ["can't be blank"]}}`
* [ ] Nenhuma das classes convertidas tem import `lombok.*`
* [ ] `./gradlew test` 100% verde

## Regras técnicas

* `ErrorResourceSerializer` é o ponto crítico: verificar o método de acesso ao campo `fieldErrors` antes de converter `ErrorResource`
* Anotações Jackson em record (`@JsonSerialize`, `@JsonIgnoreProperties`, `@JsonRootName`) são aplicadas na declaração do record — não nos componentes individuais
* Se `ErrorResource` não for candidata após análise, documentar no código: `// record inelegível: serializer customizado requer getter getFieldErrors() — ver US-06.04`

## Dependências

## Resultado esperado

`FieldErrorResource` convertida para record sem qualquer impacto funcional. `ErrorResource` convertida se a análise do serializer confirmar compatibilidade, com ajuste mínimo no `ErrorResourceSerializer` se necessário.
