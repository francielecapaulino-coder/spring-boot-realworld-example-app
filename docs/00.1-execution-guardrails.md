## Como este documento se relaciona com o docs/AGENTS.md

`docs/AGENTS.md` trata de **conformidade arquitetural** — ADRs, lista vermelha de violações de design, estrutura de pacotes. Este documento trata de algo diferente: **risco operacional** — ações destrutivas, irreversíveis ou que tocam segurança/produção, independente de qualquer ADR estar envolvido. Um agente pode estar 100% alinhado com todos os ADRs e ainda assim executar um `DROP TABLE` ou sobrescrever um `.env` de produção. É esse risco que este documento cobre.

As duas listas (`LISTA VERMELHA` do docs/AGENTS.md e os guardrails abaixo) são complementares e devem ser checadas juntas antes de qualquer execução autônoma.

## Guardrails de Execução Autônoma

### Pode executar SEM perguntar

- Leitura, criação e edição de qualquer arquivo de código
- Comandos de terminal, build, lint e testes
- Instalação de dependências (npm/pip/etc.)
- Operações de leitura no banco de dados (SELECT, queries de consulta)
- Operações de escrita no banco de dados com filtro explícito (INSERT, UPDATE/DELETE com WHERE específico)
- Git: commit, branch, diff local

### NUNCA executar sem confirmação explícita do usuário

Antes de executar qualquer item abaixo, pare e pergunte ao usuário, explicando a operação e o impacto. Não prossiga sem resposta afirmativa clara.

**Banco de dados — destrutivo ou irreversível:**
- `DROP TABLE`, `DROP DATABASE`, `DROP SCHEMA`
- `TRUNCATE`
- `UPDATE` sem cláusula `WHERE` (ou com `WHERE` que afete a tabela inteira, ex: `WHERE 1=1`)
- `DELETE` sem cláusula `WHERE`
- Migrations que removam colunas/tabelas ou alterem tipos de forma destrutiva
- Qualquer comando que altere dados diretamente em ambiente de produção

**Arquivos e configurações de segurança alta:**
- Arquivos de variáveis de ambiente (`.env`, `.env.*`)
- Arquivos de segredos, chaves, tokens e credenciais (qualquer arquivo com "secret", "key", "credential", "token" no nome ou conteúdo)
- Módulos de autenticação/autorização (ex: `/auth`, middlewares de auth, lógica de permissões)
- Configurações de infraestrutura e CI/CD (Dockerfile, docker-compose, pipelines `.github/workflows`, Terraform, configs de deploy)
- Arquivos de configuração de produção (ex: configs com sufixo `.prod`, `.production`)

### Como agir quando um guardrail for acionado

1. Pare a execução antes de rodar o comando.
2. Explique de forma direta o que o comando faria e por que está sinalizado como sensível.
3. Pergunte explicitamente: "Posso prosseguir com [ação]? (sim/não)"
4. Só execute após confirmação explícita na resposta do usuário.
5. Nunca infira confirmação de mensagens ambíguas ("ok", "continua" genérico no meio de outro contexto) — peça confirmação direcionada à ação específica.

### Regra geral

Na dúvida se uma operação é destrutiva, irreversível ou toca em segurança/produção, trate como sensível e pergunte. É preferível interromper o fluxo de vibe coding para confirmar do que executar algo irreversível silenciosamente.

## Aplicação prática no projeto RealWorld

| Padrão de arquivo no repositório | Guardrail aplicável |
|---|---|
| `.env`, `.env.example`, `.env.*` | ⚠️ Confirmação — mesmo `.env.example` sem valores reais |
| `.github/workflows/*.yml` | ⚠️ Confirmação — CI/CD |
| `docker-compose.yml`, `Dockerfile` (EPIC-03) | ⚠️ Confirmação — infraestrutura |
| `application-prod.yml` / qualquer arquivo com sufixo de produção | ⚠️ Confirmação |
| `io.spring.infrastructure.security.*`, qualquer classe de JWT/auth | ⚠️ Confirmação — módulo de autenticação |
| Migrations Flyway que removem coluna/tabela (ex: rollback de soft delete) | ⚠️ Confirmação — destrutivo |
| `docs/09-user-stories/**`, `*.md`, código de produção em `io.spring.api/application/core` (sem tocar auth) | ✅ Sem confirmação |

> Toda história de usuário deve, na seção de implementação, indicar explicitamente se algum guardrail é acionado pelos arquivos que ela toca — mesmo que a resposta seja "nenhum guardrail acionado".
